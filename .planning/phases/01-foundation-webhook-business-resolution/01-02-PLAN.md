## Phase Goal

**As a** client of a Greek service business, **I want to** text a business-specific WhatsApp deep link and be reliably routed to the right business, with a data-consent notice on my first contact, **so that** I can start a booking conversation with zero confusion about who I'm talking to, even if my message gets sent or delivered twice.

> Note: ROADMAP.md's Phase 1 `**Goal**:` line is descriptive prose, not the strict "As a / I want to / so that" format. This story is derived directly from that Goal plus Success Criteria 1-4 (business resolution, consent notice, dedup, not-found handling) without adding new scope. Success Criterion 5 (Meta Business Verification) is an operational prerequisite, not part of the client-facing story — it is covered by Plan 01-04.

<objective>
Implement the core Walking Skeleton path (SKELETON.md): a real WhatsApp webhook that verifies Meta's signature, extracts and normalizes a business code from the inbound message, looks it up against the two fixtures seeded in Plan 01-01, and sends a real Greek-language reply back through the WhatsApp Cloud API — deployed on fly.io. This satisfies Roadmap Phase 1 Success Criteria 1 and 4 (business confirmation reply; "not found" reply). Dedup and consent (Success Criteria 2 and 3) are deliberately NOT wired here — they are Plan 01-03's slice, layered on top of this same handler.

Purpose: Prove the full stack (HTTP in -> Postgres read -> HTTP out) actually works end-to-end before adding the dedup/consent refinements.
Output: A deployed (or documented locally-runnable) Express app that correctly resolves business identity from a real WhatsApp message.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Meta/WhatsApp (internet) -> Express webhook | Fully untrusted inbound HTTP; every request must be signature-verified before any processing |
| Webhook -> WhatsApp Graph API | Outbound HTTPS call carrying the access token |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-05 | Tampering | src/webhooks/whatsapp.ts POST handler | critical | mitigate | X-Hub-Signature-256 verified via `crypto.createHmac('sha256', APP_SECRET)` + `crypto.timingSafeEqual`, with an explicit buffer-length equality guard checked BEFORE the timing-safe comparison (prevents a RangeError crash on a malformed/short attacker-supplied header); unsigned/invalid requests get HTTP 403 before any JSON parsing |
| T-01-06 | Denial of Service | src/webhooks/whatsapp.ts, src/utils/validation.ts | medium | mitigate | Zod validates the full payload shape before field access; unknown/non-text message types and malformed payloads return HTTP 200/400 without throwing (Pitfall 5) |
| T-01-07 | Information Disclosure | src/server.ts error middleware | medium | mitigate | Generic 500/400 JSON bodies returned to Meta; full error + stack trace only logged server-side via pino, never included in the HTTP response |
| T-01-08 | Spoofing | src/webhooks/whatsapp.ts GET handler | low | mitigate | hub.verify_token compared against config.webhookVerifyToken before echoing hub.challenge; mismatch returns 403 |
| T-01-09 | Information Disclosure | src/whatsapp/client.ts | medium | mitigate | WHATSAPP_ACCESS_TOKEN sent only as an Authorization Bearer header over HTTPS to graph.facebook.com; never logged, never surfaced in any error returned to the client |

</threat_model>
