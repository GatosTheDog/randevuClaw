## Phase Goal

**As a** business owner using RandevuClaw, **I want to** have every confirmed booking automatically reflected on my Google Calendar, get a daily Telegram summary of today's appointments, and have my clients automatically reminded before their visit, **so that** I never manually update a calendar or chase a no-show client, and my clients show up on time without me lifting a finger.

> Note: see Plan 03-01's identical note on ROADMAP.md's Phase 3 Goal-line format. This plan delivers the "so that" clause's client-facing half: NOTF-01, appointment reminders.

<objective>
Implement the 24h-and-1h-prior client reminder sweep: an in-process poller (D-10's 15-minute interval) that finds confirmed bookings approaching their appointment time and sends each client a Telegram reminder at both thresholds, using DST-safe Athens-local arithmetic throughout and permanently skipping (never catching up) a reminder whose window had already elapsed at the moment the booking was created (D-14).

Purpose: This plan carries the phase's trickiest correctness requirement -- ROADMAP SC4 explicitly calls out DST transitions and late-night bookings as scenarios that must still land at the correct Athens local time. RESEARCH.md's Pitfall 1 (DST off-by-one) is a documented, real risk of the naive `Date.getTime()` subtraction approach; this plan's algorithm is built entirely from calendar-date-string and wall-clock-minutes arithmetic to avoid it. Per D-03, no new Telegram send helper is added -- reminders go out via the existing `sendTelegramMessage`. Per D-04, the reminder-eligibility/timing logic never touches the Telegram send call directly (it only decides WHETHER and computes the message text; the send itself is one line), keeping the channel adapter swappable. Per D-12, this poller runs in the SAME process as the Express server and the other 3 Phase 3 pollers -- no `fly.toml` `[processes]` changes.
Output: A working reminder system, self-contained in one new file plus one new line in `server.ts`.
</objective>

    `calendarDaysBetween(fromIso: string, toIso: string): number` — mirrors `addCalendarDays`'s noon-UTC-anchor technique: `Math.round((new Date(\`${toIso}T12:00:00Z\`).getTime() - new Date(\`${fromIso}T12:00:00Z\`).getTime()) / 86400000)`.
    `hadAtLeastHoursMarginAtBookingTime(booking: Booking, minHours: number): boolean` — `const createdDateIso = isoDateInAthens(booking.createdAt); const createdTimeMin = minutesSinceMidnight(athensWallClockTime(booking.createdAt)); const apptTimeMin = minutesSinceMidnight(booking.calendarTime); const daysBetween = calendarDaysBetween(createdDateIso, booking.calendarDate); if (daysBetween < 0) return false; if (daysBetween === 0) return apptTimeMin - createdTimeMin >= minHours * 60; const marginMinutes = (24 * 60 - createdTimeMin) + (daysBetween - 1) * 24 * 60 + apptTimeMin; return marginMinutes >= minHours * 60;` — this is D-14's eligibility gate, evaluated ONCE per booking from its immutable `createdAt`, independent of when the sweep runs.

    Implement one more non-exported helper: `minutesUntilAppointment(booking: Booking, now: Date): number` — `const todayIso = isoDateInAthens(now); const nowTimeMin = minutesSinceMidnight(athensWallClockTime(now)); const apptTimeMin = minutesSinceMidnight(booking.calendarTime); if (booking.calendarDate === todayIso) return apptTimeMin - nowTimeMin; const daysBetween = calendarDaysBetween(todayIso, booking.calendarDate); if (daysBetween < 0) return -Infinity; return (24 * 60 - nowTimeMin) + (daysBetween - 1) * 24 * 60 + apptTimeMin;` — a single unified "how many minutes from now until the appointment" computation (negative once the appointment has passed) that correctly handles the late-night/day-crossing case (Test 5) without separate today/tomorrow branching.
