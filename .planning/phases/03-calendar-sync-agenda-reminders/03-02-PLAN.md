## Phase Goal

**As a** business owner using RandevuClaw, **I want to** have every confirmed booking automatically reflected on my Google Calendar, get a daily Telegram summary of today's appointments, and have my clients automatically reminded before their visit, **so that** I never manually update a calendar or chase a no-show client, and my clients show up on time without me lifting a finger.

> Note: see Plan 03-01's identical note on ROADMAP.md's Phase 3 Goal-line format. This plan delivers the first "I want to" clause in full: OWNR-04, Google Calendar sync.

<objective>
Implement the Google Calendar half of Phase 3: the OAuth 2.0 consent flow (client construction, auth URL, code exchange), the Calendar CRUD service (create/update/delete events, best-effort and non-blocking per D-15), the one-time CLI setup tool the project owner runs per fixture business (D-05/D-07), the retry poller for failed syncs (D-16), and the three integration hooks into the existing booking-confirmation/cancellation/reschedule-approval code paths built in Phase 2.

Purpose: This is the highest-risk plan in the phase — it is the only one that talks to an external API with real user credentials (an OAuth refresh token) and the only one where a coding mistake could either leak a client's booking into the wrong business's calendar or silently corrupt booking state on an API failure. D-15's "best-effort, non-blocking" contract and the OAuth `state` CSRF guard (this plan's own addition, not explicitly in RESEARCH.md — see Task 2) are the two guardrails that make this safe. Per D-04, `src/calendar/sync.ts`'s Calendar-API logic never touches Telegram at all (it is pure Google-API business logic, called FROM the Telegram-facing integration points, not the other way around) — a channel-agnostic split matching Phase 2's own adapter boundary. Per D-12, the retry poller runs in the SAME process as the Express server and the other Phase 3 pollers, started from `server.ts` — no `fly.toml` `[processes]` changes.
Output: A fully working, best-effort Google Calendar mirror of the `bookings` table for any business with a stored refresh token, self-healing via the retry poller.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application -> Google Calendar API | Outbound only; the refresh token grants read/write access to exactly one business owner's calendar, scoped by which business row the sync call reads its `googleRefreshToken` from |
| local browser -> scripts/setup-google-calendar.ts loopback HTTP server | The OAuth authorization-code callback; any local process (not just the real browser redirect) could in principle hit this port first |
| Telegram callback_query / client cancel -> Calendar sync hooks | Both are already-authenticated, already-authorized code paths (Phase 2's owner-identity check and client-ownership check run BEFORE these hooks fire) -- Calendar sync introduces no new authorization decision, it only mirrors an already-approved state transition |

| T-03-04 | Spoofing | scripts/setup-google-calendar.ts OAuth callback | high | mitigate | A random `state` CSRF token is generated per run and embedded in the auth URL (`getOAuth2AuthUrl(state)`); the loopback callback handler rejects any request whose `state` query param does not match exactly, before ever exchanging the `code` -- prevents authorization-code injection from a different, attacker-initiated OAuth flow |
