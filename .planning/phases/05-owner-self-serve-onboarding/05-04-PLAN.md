    Create src/webhooks/platform.ts. Export one async function: handlePlatformBotWebhook(req: Request, res: Response): Promise void.

    The handler follows this structure:

    HMAC verification (lines 1-20): Read x-telegram-bot-api-secret-token header. Use crypto.timingSafeEqual(Buffer.from(headerValue ?? ''), Buffer.from(config.platformWebhookSecret)) with try/catch — on mismatch or catch: logger.warn and return res.status(401).send('Unauthorized').

    Update parsing: Parse req.body as TelegramUpdate (reuse interface from telegram.ts or define inline). Extract update_id, message (if present), callback_query. If neither message nor callback_query: return res.status(200).send('OK') with a logger.info for unsupported type.

    Extract ownerTelegramId: from message?.from?.id or callback_query?.from?.id, stringify. Extract updateId: String(update.update_id). Extract messageText: message?.text?.trim() ?? ''.

    Wrap remaining logic in: try { res.status(200).send('OK'); await botTokenStore.run(config.platformBotToken, async () => { ... }); } catch (err) { logger.error(err, '...'); } finally { if (!res.headersSent) res.status(200).send('OK'); }.

    Inside botTokenStore.run:
    1. Dedup: const dedupResult = await insertOrIgnoreTelegramUpdate(updateId, null, ownerTelegramId, 'message'). If 'ignored': return.

    2. Look up active session: const activeResult = await findActiveSessionByOwnerTelegramId(ownerTelegramId).

    3. Branch:
       A. If activeResult is non-null (session exists and not 'done'): call dispatchOnboardingStep(activeResult.session, activeResult.business, ownerTelegramId, messageText). Return.

       B. If no active session: check for a completed business — const existingBusiness = await findBusinessByOwnerTelegramId(ownerTelegramId).
          B1. If existingBusiness exists (step was 'done' — re-registration): Treat messageText as new bot token. Call getMeBotInfo(messageText.trim()) to validate. On success: call unregisterBotWebhook(existingBusiness.botToken!) to clear old webhook. Generate new webhookId = crypto.randomUUID(), webhookSecret = crypto.randomBytes(32).toString('hex'). Call activateBusiness(existingBusiness.id, webhookId, webhookSecret) to update businesses table with new credentials. Also update businesses.botToken to the new token (direct db.update call). Call createOrResetOnboardingSession(existingBusiness.id, 'name') to reset flow. Send: 'Το παλιό bot αποσυνδέθηκε. Πώς ονομάζεται η επιχείρησή σας;'.
          B2. If existingBusiness is null (brand new owner): Treat messageText as new bot token. Call getMeBotInfo(messageText.trim()). On getMe failure: send Greek error 'Μη έγκυρο token bot. Παρακαλώ ελέγξτε και ξαναστείλτε.' and return. On success: generate webhookId = crypto.randomUUID(), webhookSecret = crypto.randomBytes(32).toString('hex'). Generate placeholder slug using a timestamp suffix (e.g. 'business-' + Date.now()). Call createBusinessForOnboarding({ ownerTelegramId, name: 'Νέα Επιχείρηση', slug: placeholderSlug, botToken: messageText.trim(), webhookId, webhookSecret }). Call createOrResetOnboardingSession(newBusiness.id, 'name'). Send: 'Καλωσήρθατε! Πώς ονομάζεται η επιχείρησή σας;'.

    Import config from src/config.ts. Import crypto from 'crypto'. Do NOT import or call getOrCreateBotInstance — the platform bot does not use Telegraf middleware.

    All logger calls use pino logger from src/utils/logger.ts. Never log ownerTelegramId in error paths — use it only in info/debug level with explicit intent.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram → platform handler | Every incoming webhook must pass HMAC check; all other requests rejected 401 |
| owner message text → getMe API | Owner-submitted bot token sent to Telegram for validation; must not be logged |
| platform handler → DB | Creates business and session rows; uses admin db (cross-tenant by design) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-10 | Spoofing | HMAC bypass on platform webhook endpoint | high | mitigate | crypto.timingSafeEqual with PLATFORM_WEBHOOK_SECRET; same battle-tested pattern as per-business bot handler in Phase 4 |
| T-05-11 | Repudiation | Duplicate platform update processed twice | medium | mitigate | insertOrIgnoreTelegramUpdate dedup-insert with unique updateId; 'ignored' result short-circuits handler |
| T-05-12 | Information Disclosure | Owner's new bot token logged during getMe call | high | mitigate | getMeBotInfo logs { method: 'getMe' } only (callTelegramApiDirect pattern from Plan 02 — no token logging) |
| T-05-13 | Elevation of Privilege | Arbitrary user creates a business for another owner's Telegram ID | medium | mitigate | Ownership anchored to from.id at INSERT time (D-02); no separate ownership-claim step; only the sender can become the owner |
| T-05-14 | Denial of Service | Flooding platform with invalid bot tokens (creates junk business rows) | low | accept | PoC scope; rate limiting not in scope; UNIQUE constraint on telegram_updates.updateId and businesses.ownerTelegramId (one business per ownerTelegramId via existing business lookup) limits blast radius |
| T-05-SC | Tampering | npm installs | low | accept | No new packages |
</threat_model>
