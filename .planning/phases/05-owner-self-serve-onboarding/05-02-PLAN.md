## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application → Telegram API | callTelegramApiDirect sends the owner's bot token in the HTTP URL; must never appear in logs |
| platform handler → DB | onboarding/queries.ts uses admin db bypassing RLS — intentional for cross-tenant session lookups |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-03 | Information Disclosure | Bot token in callTelegramApiDirect URL logged | high | mitigate | logger.debug({ method }, ...) logs method name only; botToken parameter never passed to logger; same pattern as existing callTelegramApi (line 31 of client.ts) |
| T-05-04 | Spoofing | getMeBotInfo accepts any string as bot token without format validation | low | accept | Telegram's getMe endpoint returns 401 for invalid tokens; application-level error thrown; no need for pre-validation regex |
| T-05-05 | Tampering | createOrResetOnboardingSession resets active session for existing owner | medium | mitigate | Session reset only happens when createOrResetOnboardingSession is called from platform handler's registration path — requires a valid getMe() call first; platform handler validates token before invoking this |
| T-05-SC | Tampering | npm installs | low | accept | No new packages; all imports are project-existing (drizzle-orm, pg, crypto already in package.json) |
</threat_model>
