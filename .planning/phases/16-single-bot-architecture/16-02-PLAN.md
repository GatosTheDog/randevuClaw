## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram → business bot webhook | Untrusted sender ID from update.message.from.id (trusted only to Telegram's API signature, already verified before reaching handleFoundBusiness) |
| business.ownerTelegramId → routing decision | DB-sourced; not controllable by the sender — safe as auth anchor for PoC |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-04 | Spoofing | handleFoundBusiness ownerTelegramId comparison | high | mitigate | Explicit null guard (business.ownerTelegramId !== null AND ===) prevents null from matching; Telegram API enforces sender identity via HMAC-verified webhook secret already applied upstream. |
| T-16-05 | Tampering | onboarding_completed flag — can a client flip it? | low | accept | Flag is only written by the terminal step handler in steps.ts, which is invoked only when the owner is routing through the onboarding path. Clients are routed to routeConversationMessage and never touch the businesses table. |
| T-16-06 | Information Disclosure | Session data logged during onboarding dispatch | low | mitigate | dispatchOnboardingStep and sendTelegramMessage follow existing log hygiene: only step name and ownerTelegramId are logged, never session collectedData or bot token. |
| T-16-07 | Elevation of Privilege | Client masquerades as admin (sends ownerTelegramId) | high | mitigate | Telegram user IDs are assigned by Telegram and cannot be forged by end users; the HMAC webhook signature (already verified in handleTelegramWebhookPost) ensures the update originates from Telegram's servers. |
</threat_model>
