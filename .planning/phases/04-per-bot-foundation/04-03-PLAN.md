<objective>
Refactor the query layer and Telegram client to thread RLS context and per-request bot token transparently through AsyncLocalStorage.

Purpose: This is the core infrastructure that makes BOT-05 (RLS enforcement) work in practice. WithBusinessContext opens an appDb transaction with SET LOCAL, and every query function within that transaction sees the RLS policy. The botTokenStore makes callTelegramApi use the per-request bot token without changing any function signatures (D-02 honored). Plans 04-04 and 04-05 depend on these two functions being available.

Output: queries.ts with AsyncLocalStorage tx threading and two new functions; client.ts with botTokenStore replacing config.telegramBotToken.
</objective>

PART B — AsyncLocalStorage helper (add after existing imports, before the Business interface):
```
// Thread the current Drizzle transaction through the call stack transparently.
// Within withBusinessContext, queries use the appDb transaction (RLS enforced).
// Outside withBusinessContext (pollers, routing lookups), queries fall back to
// admin db (superuser) which bypasses RLS — this is intentional for cross-tenant ops.
const currentTx = new AsyncLocalStorage<typeof db>();

function getConn(): typeof db {
  return currentTx.getStore() ?? db;
}
```

`withBusinessContext<T>(businessId: string | number, callback: () => Promise<T>): Promise<T>` — implements D-10:
- Opens `appDb.transaction(async (tx) => { ... })`
- First statement inside: `await tx.execute(sql\`SET LOCAL app.current_business_id = ${String(businessId)}\`)`
- Then: `return currentTx.run(tx as unknown as typeof db, callback);`
- The `currentTx.run` makes `tx` available to all query functions called within `callback` via `getConn()`
