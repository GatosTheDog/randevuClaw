# Phase 4: Per-Bot Foundation - Context

**Gathered:** 2026-07-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate the Telegram layer to Telegraf, add per-token webhook routing with a UUID-keyed routing URL, implement per-bot HMAC secret verification with constant-time comparison, and enforce PostgreSQL Row Level Security at the database layer — every v1.1 feature builds on this infrastructure.

**Requirements in scope:** BOT-02, BOT-03, BOT-04, BOT-05
**Out of scope:** Owner self-serve onboarding (Phase 5), GDPR deletion (Phase 6), Gemini rate-limit resilience (Phase 6), any WhatsApp work (v1.2+)

</domain>

<decisions>
## Implementation Decisions

### D1: Telegraf Integration Boundary

- **D-01:** Telegraf is a **webhook adapter only**. It handles inbound webhook parsing and update routing (replaces manual `TelegramUpdate` JSON parsing + interface types). Existing `handleFoundBusiness`, `routeConversationMessage`, `sendTelegramMessage`, `answerCallbackQuery`, and `editTelegramMessageReplyMarkup` handlers stay unchanged.
- **D-02:** Outbound Telegram API calls keep our raw `callTelegramApi` wrappers. No `ctx.reply` or `ctx.telegram` usage — send-path tests require zero changes.
- **D-03:** **One Telegraf instance per registered bot**, stored in a `Map<webhookId, Telegraf>`. The webhook handler at `/webhooks/telegram/:webhookId` looks up the correct Telegraf instance and calls `.handleUpdate(update)`. Clean tenant isolation — no shared instance state across bots.

### D2: Webhook Routing Key & HMAC Security

- **D-04:** Route is `/webhooks/telegram/:webhookId` where `webhookId` is a **random UUID** stored in `businesses.webhook_id`. The actual bot token **never appears in the URL or logs** — satisfies STATE.md blocker: "bot token must never appear in logs or URL paths."
- **D-05:** HMAC secret per bot is generated by the **platform** via `crypto.randomBytes(32).toString('hex')` at registration/seeding time. Stored in `businesses.webhook_secret`. Telegram's `setWebhook` is called with `secret_token=` that value.
- **D-06:** Webhook secret verification uses **`crypto.timingSafeEqual`** immediately in Phase 4 (not deferred). All incoming requests on the `:webhookId` route are rejected with 401 if `X-Telegram-Bot-Api-Secret-Token` header doesn't match.

### D3: Schema Additions

- **D-07:** **Drizzle migration 0003** adds three nullable columns to `businesses`: `bot_token TEXT`, `webhook_id TEXT UNIQUE`, `webhook_secret TEXT`. Nullable follows the established convention (see schema.ts comments for `ownerTelegramId` and `googleRefreshToken`).
- **D-08:** Global config vars `TELEGRAM_BOT_TOKEN` and `TELEGRAM_WEBHOOK_SECRET` are **removed** from `config.ts`. All bot token/secret config is DB-driven. No backward-compat fallback.
- **D-09:** Phase 4 test bots are seeded from env vars: `TEST_BOT_1_TOKEN`, `TEST_BOT_1_WEBHOOK_SECRET`, `TEST_BOT_1_WEBHOOK_ID`, `TEST_BOT_2_TOKEN`, `TEST_BOT_2_WEBHOOK_SECRET`, `TEST_BOT_2_WEBHOOK_ID`. `seed.ts` populates `businesses` rows from these vars. The 208 existing tests must be updated to use this new seeded structure.

### D4: RLS Enforcement

- **D-10:** RLS context is set via **`SET LOCAL app.current_business_id = ?`** at the top of a transaction wrapper that surrounds all conversation-handling DB operations. Scope: per-transaction (auto-cleared on commit/rollback). No session-level leakage via connection pool reuse.
- **D-11:** A dedicated **`randevuclaw_app` Postgres role** (non-superuser) is created with SELECT/INSERT/UPDATE/DELETE on all business tables. RLS policies use `current_setting('app.current_business_id', true)`. The app connects as this role — superuser connection is only for migrations.
- **D-12:** RLS policies apply to **all business-scoped tables**: `messages`, `client_business_relationships`, `bookings`, `services`, `availability`, `telegram_updates`, `calendar_sync_queue`. The `businesses` table itself gets a policy so a business's Telegraf instance can only read its own row.

### Claude's Discretion

- Exact Telegraf version to install (4.15+ per BOT-04; latest stable preferred).
- How to structure the `Map<webhookId, Telegraf>` registry — module-level singleton vs class vs factory function.
- Transaction wrapper shape — utility function wrapping Drizzle's `db.transaction()` or a thin helper that calls `SET LOCAL` before yielding.
- Order of migration execution (apply 0003 to Neon live DB + local test DB simultaneously, matching the established pattern from Phase 3).

</decisions>

<specifics>
## Specific Ideas

- "One Telegraf instance per bot in a Map" — the user explicitly preferred this over a single shared Telegraf instance.
- UUID routing key (not slug, not raw token) — explicitly chosen to satisfy the STATE.md security blocker.
- Remove global telegram env vars completely — no backward-compat fallback shim.

</specifics>
