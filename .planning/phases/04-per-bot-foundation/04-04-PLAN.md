PART C — Replace handleTelegramWebhookPost (lines 200-276 in the current file) with the refactored implementation:

The new handleTelegramWebhookPost must perform these steps in order:

Step 1 — Extract webhookId:
Read `req.params.webhookId`. If absent or empty, return 400 with body 'Bad Request'.

Step 2 — Business lookup (pre-auth, admin db — per D-04):
Call `findBusinessByWebhookId(webhookId)`. If the result is null, or if `business.webhookSecret` or `business.botToken` is falsy, log `logger.warn({ webhookId }, 'Webhook ID not found or bot credentials incomplete')` and return 404. Never log business.botToken or business.webhookSecret — log only webhookId.

Step 3 — Constant-time HMAC verification (per D-06):
`const headerValue = req.headers['x-telegram-bot-api-secret-token'] as string | undefined;`
`const headerBuffer = Buffer.from(headerValue ?? '');`
`const secretBuffer = Buffer.from(business.webhookSecret);`
Declare `let secretValid: boolean`. Wrap in try/catch: inside try, assign `secretValid = crypto.timingSafeEqual(headerBuffer, secretBuffer)`; in catch block (fires when buffers have different lengths), set `secretValid = false`. If `!secretValid`, log `logger.warn({ webhookId }, 'Webhook secret verification failed')` and return 401. Never log the secret value.

Step 4 — Bot instance and update parsing:
`const update = req.body as TelegramUpdate;`
`const bot = getOrCreateBotInstance(webhookId, business.botToken);`

Step 5 — Per-request context + existing handler pipeline:
Wrap the remaining processing in `await botTokenStore.run(business.botToken, async () => { await withBusinessContext(business.id, async () => { [INNER LOGIC] }); });`

Inside the inner async callback, run the existing dedup + routing logic — this is the only part of the current handler body that's retained. Use `business` (already resolved) as the context instead of slug extraction. Specifically:

a) `const updateId = String(update.update_id);`
b) `const senderTelegramId = String(update.message?.from.id ?? update.callback_query?.from.id ?? '');`
c) `const updateType = update.message ? 'message' : 'callback_query';`
d) `logger.info({ updateId, webhookId, senderTelegramId, updateType }, 'Telegram update received');` — log webhookId not botToken
e) `const dedupResult = await insertOrIgnoreTelegramUpdate(updateId, business.id, senderTelegramId, updateType);` — pass `business.id` (not null as before); this also satisfies the RLS context set by withBusinessContext
f) If `dedupResult === 'ignored'`: `logger.info({ updateId }, 'Duplicate Telegram update ignored'); return;`
g) Call `await bot.handleUpdate(update);` to satisfy BOT-04 (Telegraf as webhook adapter per D-03). Since no middleware is attached in Phase 4, this validates the update structure and is a no-op for dispatch. The actual handler dispatch below is explicit.
h) If `update.callback_query`: `await handleCallbackQuery(update.callback_query, senderTelegramId);` then `return;`
i) If `update.message`: `await handleFoundBusiness(updateId, business, senderTelegramId, update.message.text ?? '');`

The business object from findBusinessByWebhookId already identifies the tenant — slug extraction (extractAndNormalizeAllBusinessCodeCandidates, findBusinessBySlug, findLatestBusinessForClient) is NOT called from the main handler path. The `business` parameter in handleFoundBusiness is the one retrieved from the webhookId lookup.

Note: handleFoundBusiness internally calls routeConversationMessage which performs conversation logic independently of how business was resolved. This satisfies D-01 (handler unchanged).

After the botTokenStore.run call, send `res.status(200).send('OK')`. Wrap the entire function body in try/catch/finally as before: in catch, `logger.error({ err }, 'Telegram webhook handler failed');`; in finally, `if (!res.headersSent) res.status(200).send('OK');`. Telegram retries if it receives non-2xx — always respond 200 on success or handled error.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram→Express | Incoming webhook POST arrives from internet; webhookId in URL is attacker-controlled input until HMAC passes |
| Express→appDb | After HMAC passes, withBusinessContext sets RLS tenant; any DB call outside this context uses admin db (intended for pre-auth lookups only) |
| botTokenStore→callTelegramApi | Bot token flows from business.botToken through AsyncLocalStorage to Telegram API URL; never appears in logs or response bodies |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-09 | Spoofing | POST /webhooks/telegram/:webhookId | high | mitigate | findBusinessByWebhookId lookup (404 on unknown webhookId) followed by crypto.timingSafeEqual check (401 on invalid secret) before any update processing begins |
| T-04-10 | Tampering | HMAC verification in handleTelegramWebhookPost | high | mitigate | crypto.timingSafeEqual used exclusively — string equality (`===`) is forbidden; acceptance criterion explicitly greps for timingSafeEqual in the source file |
| T-04-11 | Information Disclosure | Bot token in logs or URL | high | mitigate | URL path uses webhookId (UUID, opaque); all logger calls in new handler pass only `{ webhookId, updateId, senderTelegramId, updateType, err }` — botToken and webhookSecret never appear as log fields; pino redaction from Plan 04-02 is defense-in-depth |
| T-04-12 | Elevation of Privilege | Cross-tenant DB access if withBusinessContext is skipped | critical | mitigate | All DB calls inside handleFoundBusiness and handleCallbackQuery run within withBusinessContext (which sets SET LOCAL app.current_business_id); RLS policies at DB layer enforce isolation even if app code has a bug |
| T-04-13 | Spoofing | Attacker crafts a valid UUID as webhookId guessing | low | accept | UUID v4 has 2^122 key space; practical enumeration is infeasible; each UUID still requires matching HMAC secret to pass Step 3 |
| T-04-SC | Tampering | npm/pip/cargo installs | low | accept | No new npm packages in this plan; all imports come from packages installed in Plans 04-01/04-02 (telegraf, crypto built-in) |
