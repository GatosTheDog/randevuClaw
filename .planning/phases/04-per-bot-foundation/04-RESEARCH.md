# Phase 4: Per-Bot Foundation - Research

**Researched:** 2026-07-10  
**Domain:** Telegram bot infrastructure migration, per-tenant webhook routing, database-layer multi-tenant isolation  
**Confidence:** HIGH

## Summary

This phase bridges from a single monolithic bot to a per-business bot architecture without rewriting existing conversation logic. Telegraf replaces only the inbound webhook parsing layer (not the outbound API calls), a UUID-keyed routing layer adds per-bot isolation at the HTTP layer, and PostgreSQL Row Level Security (RLS) enforces isolation at the database layer—ensuring that even an application code bug that omits a WHERE clause cannot leak data across tenants.

The 208 existing tests will continue to pass unchanged because the conversation handlers (`routeConversationMessage`, `handleFoundBusiness`, callback-query logic) remain identical; only the webhook adapter and routing change. Per-bot HMAC secret verification using `crypto.timingSafeEqual` prevents webhook replay attacks and cross-tenant message hijacking.

**Primary recommendation:** Use Telegraf 4.15+ in webhook-adapter-only mode (no `ctx.reply` or `ctx.telegram`), store one `Telegraf` instance per webhook ID in a Map, route via `/webhooks/telegram/:webhookId` where `webhookId` is a UUID, and enforce RLS at the database layer with transaction-scoped `SET LOCAL` context variables.

### Locked Decisions

- **D-01:** Telegraf is a **webhook adapter only**. Existing `handleFoundBusiness`, `routeConversationMessage`, `sendTelegramMessage`, `answerCallbackQuery`, and `editTelegramMessageReplyMarkup` handlers stay unchanged.
- **D-02:** Outbound Telegram API calls keep raw `callTelegramApi` wrappers. No `ctx.reply` or `ctx.telegram` usage — send-path tests require zero changes.
- **D-03:** **One Telegraf instance per registered bot**, stored in a `Map<webhookId, Telegraf>`. The webhook handler at `/webhooks/telegram/:webhookId` looks up the correct Telegraf instance and calls `.handleUpdate(update)`.
- **D-04:** Route is `/webhooks/telegram/:webhookId` where `webhookId` is a **random UUID** stored in `businesses.webhook_id`. The actual bot token **never appears in the URL or logs**.

- **D-06:** Webhook secret verification uses **`crypto.timingSafeEqual`** immediately in Phase 4 (not deferred). All incoming requests on the `:webhookId` route are rejected with 401 if `X-Telegram-Bot-Api-Secret-Token` header doesn't match.
- **D-07:** **Drizzle migration 0003** adds three nullable columns to `businesses`: `bot_token TEXT`, `webhook_id TEXT UNIQUE`, `webhook_secret TEXT`.
- **D-08:** Global config vars `TELEGRAM_BOT_TOKEN` and `TELEGRAM_WEBHOOK_SECRET` are **removed** from `config.ts`. All bot token/secret config is DB-driven. No backward-compat fallback.
- **D-09:** Phase 4 test bots are seeded from env vars: `TEST_BOT_1_TOKEN`, `TEST_BOT_1_WEBHOOK_SECRET`, `TEST_BOT_1_WEBHOOK_ID`, `TEST_BOT_2_TOKEN`, `TEST_BOT_2_WEBHOOK_SECRET`, `TEST_BOT_2_WEBHOOK_ID`. `seed.ts` populates `businesses` rows from these vars. The 208 existing tests must be updated to use this new seeded structure.
- **D-10:** RLS context is set via **`SET LOCAL app.current_business_id = ?`** at the top of a transaction wrapper that surrounds all conversation-handling DB operations. Scope: per-transaction (auto-cleared on commit/rollback). No session-level leakage via connection pool reuse.
- **D-11:** A dedicated **`randevuclaw_app` Postgres role** (non-superuser) is created with SELECT/INSERT/UPDATE/DELETE on all business tables. RLS policies use `current_setting('app.current_business_id', true)`. The app connects as this role — superuser connection is only for migrations.
- **D-12:** RLS policies apply to **all business-scoped tables**: `messages`, `client_business_relationships`, `bookings`, `services`, `availability`, `telegram_updates`, `calendar_sync_queue`. The `businesses` table itself gets a policy so a business's Telegraf instance can only read its own row.

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| **Telegraf webhook mode** | node-telegram-bot-api (polling) | Polling is inefficient; webhooks receive messages instantly. Telegraf webhook mode is optimized for serverless (fly.io). |

**Data flow:**
1. Telegram webhook arrives at `/webhooks/telegram/:webhookId`
2. Express middleware verifies `X-Telegram-Bot-Api-Secret-Token` header using constant-time comparison
3. Webhook ID is looked up in `businesses` table to fetch `bot_token` and `businessId`
4. Correct Telegraf instance retrieved from registry by `webhookId`
5. `Telegraf.handleUpdate(update)` extracts message/callback_query
6. Handler (`handleFoundBusiness`, `handleCallbackQuery`) is invoked
7. All DB operations wrapped in `withBusinessContext(businessId)` which sets `SET LOCAL app.current_business_id`
8. RLS policies at PostgreSQL layer filter rows to current business only
9. Response queries handler (e.g., `findServiceById`, `findBookingByIdUnscoped`) and business context allows re-derived checks
10. Outbound message sent via existing `callTelegramApi` wrappers with bot token from database

### Pattern 1: Per-Bot Telegraf Registry

**What:** One `Telegraf` instance per registered bot token, indexed by webhook ID in a memory-resident `Map`. Incoming webhooks look up the correct instance by ID.

**When to use:** Multi-tenant bot architectures where each tenant has their own bot token and webhook. Eliminates shared state between bots (context pollution, race conditions).

**Example:**

```typescript
// src/telegram/registry.ts
import { Telegraf } from 'telegraf';

type BotRegistry = Map<string, Telegraf>;
const botRegistry: BotRegistry = new Map();

/**
 * Retrieve or create a Telegraf instance for a given webhookId.
 * In production, call this once per registered bot at startup/registration time.
 * Tests create instances on-demand via seed data.
 */
export function getOrCreateBotInstance(webhookId: string, botToken: string): Telegraf {
  if (botRegistry.has(webhookId)) {
    return botRegistry.get(webhookId)!;
  }
  const bot = new Telegraf(botToken);
  // In webhook mode, do NOT call bot.launch() — the webhook handler calls .handleUpdate()
  botRegistry.set(webhookId, bot);
  return bot;
}

export function clearBotRegistry(): void {
  botRegistry.clear();
}
```

Source: [Telegraf webhook mode guide](https://telegraf.js.org/) + [RandevuClaw architecture](./04-CONTEXT.md#D3-Schema-Additions)

### Pattern 2: Transaction-Scoped RLS Context

**What:** Wrap all DB operations in a transaction that sets `SET LOCAL app.current_business_id = ?` before any query runs. PostgreSQL RLS policies filter rows based on this context variable.

**When to use:** Multi-tenant databases where row-level isolation must be enforced at the database layer, not just application code.

**Example:**

```typescript
// src/database/queries.ts (excerpt)
import { db } from './client'; // Drizzle client

/**
 * Execute a callback within a transaction with RLS context set.
 * Automatically clears context on commit/rollback (transaction-scoped).
 */
export async function withBusinessContext<T>(
  businessId: string | number,
  callback: () => Promise<T>
): Promise<T> {
  return db.transaction(async (tx) => {
    // Set the context variable for RLS policies
    await tx.execute(sql`SET LOCAL app.current_business_id = ${String(businessId)}`);
    // All subsequent queries in this transaction see the context
    return callback();
  });
}

// Usage:
async function handleIncomingMessage(businessId: number, senderTelegramId: string, text: string) {
  return withBusinessContext(businessId, async () => {
    // All queries below are automatically filtered by RLS
    const messages = await db.select().from(messagesTable).where(eq(messagesTable.businessId, businessId));
    const bookings = await db.select().from(bookingsTable);
    // ^^^ Even though this query has no .where() clause, RLS policies filter to current business
    return bookings;
  });
}
```

Source: [PostgreSQL RLS with Drizzle](https://orm.drizzle.team/docs/rls) + [CONTEXT.md D-10](./04-CONTEXT.md#D4-RLS-Enforcement)

### Pattern 3: HMAC Secret Verification with Constant-Time Comparison

**What:** Verify the `X-Telegram-Bot-Api-Secret-Token` header against the stored webhook secret using `crypto.timingSafeEqual` to prevent timing attacks.

**When to use:** Any webhook that receives authentication tokens. Timing attacks allow attackers to reconstruct secrets byte-by-byte by measuring request latency.

**Example:**

```typescript
// src/webhooks/telegram.ts (excerpt)
import crypto from 'crypto';
import { findBusinessByWebhookId } from '../database/queries';

/**
 * Verify Telegram webhook secret token using constant-time comparison.
 * Prevents timing attacks: comparison always takes the same duration
 * regardless of where the bytes differ.
 */
export async function verifyAndRouteTelegramWebhook(req: Request, res: Response): Promise<void> {
  try {
    const headerValue = req.headers['x-telegram-bot-api-secret-token'] as string | undefined;
    const webhookId = req.params.webhookId;

    if (!webhookId) {
      res.status(400).send('Bad Request: missing webhookId');
      return;
    }

    // Look up the business and its webhook secret
    const business = await findBusinessByWebhookId(webhookId);
    if (!business || !business.webhookSecret) {
      res.status(404).send('Not Found');
      return;
    }

    // Constant-time comparison: prevents timing attacks
    const headerBuffer = Buffer.from(headerValue ?? '');
    const secretBuffer = Buffer.from(business.webhookSecret);

    try {
      crypto.timingSafeEqual(headerBuffer, secretBuffer);
    } catch (err) {
      // timingSafeEqual throws if buffers have different lengths
      res.status(401).send('Unauthorized');
      return;
    }

    // Secret is valid; proceed to message handling
    const update = req.body as TelegramUpdate;
    const bot = getOrCreateBotInstance(webhookId, business.botToken);
    await bot.handleUpdate(update);

    res.status(200).send('OK');
  } catch (err) {
    logger.error({ err }, 'Telegram webhook handler failed');
    res.status(200).send('OK'); // Always 200 to Telegram, never retry
  }
}
```

Source: [Node.js crypto.timingSafeEqual](https://nodejs.org/api/crypto.html#crypto_crypto_timingsafeequal_a_b) + [Telegram webhook security](https://core.telegram.org/bots/api#setwebhook)

### Anti-Patterns to Avoid

- **String equality for secrets** (`headerValue === secret`): Leaks timing information. Always use `crypto.timingSafeEqual`.
- **Shared Telegraf instance across bots** (`const bot = new Telegraf(token)` once globally): Each bot's middleware/handlers run in the same instance, risking context pollution and race conditions. Use per-bot registry.
- **Webhook secret in logs**: Token exposure in structured logs. Redact bot tokens and webhook secrets from all logging; log only the `webhookId` (UUID).
- **Webhook secret in URL**: Ever put the raw bot token or secret in a URL path (`/webhooks/telegram/${botToken}`). Use opaque UUID (`/webhooks/telegram/${webhookId}`).
- **RLS policies as optional**: Enforce RLS at the database layer even if application code looks "careful" about filtering. A single missing WHERE clause becomes a data leak without RLS.
- **Session-level RLS context**: Setting `SET app.current_business_id` at the session level leaks context across requests if connections are pooled. Use `SET LOCAL` (transaction-scoped) only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| **HMAC verification** | Manual byte comparison or simple equality | `crypto.timingSafeEqual` | Timing attacks. Attackers reconstruct secrets by measuring latency. Only constant-time comparison is safe. |
| **Telegram webhook parsing** | Manual JSON parsing + type coercion | Telegraf SDK `.handleUpdate()` | Handles API versioning, field deprecations, type safety. Reduces bugs from manual extraction. |
| **Per-bot state management** | Custom class or mutable object | `Map<webhookId, Telegraf>` | Simple, thread-safe (no async state mutations), familiar pattern. |
| **Row-level security** | Application-level WHERE clauses only | PostgreSQL RLS policies + transactions | Defense-in-depth. A single forgotten WHERE clause becomes a leak without RLS. RLS is the hard boundary. |
| **Bot token storage** | Environment variables or plaintext | Database columns (`bot_token`, `webhook_secret`) with RLS | Allows per-business tokens, rotation, audit trails. Environment variables don't scale to N tenants. |
| **Webhook routing** | String-matching on bot token | UUID lookup in database | Opaque routing key; token never exposed in logs, URLs, or proxies. |

**Key insight:** Multi-tenancy cannot be bolted on with application logic alone. The database, infrastructure, and authentication layer must all enforce isolation — the "Swiss cheese" model (multiple layers of defense) is the industry standard for SaaS.

## Common Pitfalls

### Pitfall 1: Bot Token Leakage in Logs or URLs

**What goes wrong:** A log statement or error message accidentally includes the raw bot token or webhook secret. Attacker gains a working bot token and can hijack conversations.

**Why it happens:** Tokens are short strings, easy to accidentally log during debugging. URLs show in browser history, reverse proxy logs, and monitoring dashboards.

**How to avoid:** 
- Use UUID (`webhookId`) in URL path, not token.
- Redact `botToken` and `webhookSecret` from all logging. Log only `webhookId`.
- Test that logs never include tokens: grep test output for token values.
- Use structured logging with field redaction (pino has built-in `.redact()` support).

**Warning signs:** 
- A log line includes the actual bot token string.
- URLs in reverse proxy logs show the bot token or secret.
- Error messages from `callTelegramApi` reveal token.

### Pitfall 2: Cross-Tenant Data Access via Missing RLS Policy

**What goes wrong:** A developer adds a new table (e.g., `announcements`) but forgets the RLS policy. A query omits the `WHERE business_id = ?` clause, accidentally returning rows from all businesses.

**Why it happens:** RLS is invisible until a policy is missing. A query that works in single-tenant development can leak data when multi-tenant.

**How to avoid:**
- Add RLS policies **at migration time**, not as a post-hoc fix.
- Test every table's RLS policy explicitly: create two test businesses, insert rows, query without WHERE clause, verify only current business rows are returned.
- Define a checklist: "new table → new RLS policy → new test case".
- Run the full test suite (208 tests) to catch policy bugs.

**Warning signs:**
- Test assertion: "expected 1 booking, got 2" (the 2nd is from another tenant).
- A new table added without a corresponding migration RLS policy.

### Pitfall 3: Timing Attack on HMAC via String Equality

**What goes wrong:** A webhook secret is verified using simple string equality (`===`). An attacker guesses the secret byte-by-byte by measuring request latency.

**Why it happens:** String comparison short-circuits as soon as it finds a mismatch. First byte wrong = 0.1µs. First 30 bytes right, last byte wrong = 3µs. Attacker measures latency and reconstructs the secret.

**How to avoid:** 
- Use `crypto.timingSafeEqual(Buffer.from(headerValue), Buffer.from(storedSecret))` always.
- Add a test: compare a valid secret (should succeed) and an invalid secret (should fail), verify the function doesn't short-circuit.

**Warning signs:**
- Webhook verification uses `===` or `.equals()`.
- No explicit unit test for timing-safe comparison.

### Pitfall 4: Transaction Context Leakage via Session-Level Variables

**What goes wrong:** RLS context is set at session level (`SET app.current_business_id = ?` without LOCAL). Connection pooling reuses the session for the next request, leaking the old business ID to the new tenant.

**Why it happens:** `SET LOCAL` is transaction-scoped (auto-clears on commit). `SET` without LOCAL is session-scoped (persists across connections from the pool).

**How to avoid:**
- Use `SET LOCAL` in every transaction wrapper. Never use `SET` without LOCAL.
- Add a test: execute two sequential transactions with different business IDs, verify rows from each are isolated.

**Warning signs:**
- `SET app.current_business_id = ?` (without LOCAL keyword).
- Test flakiness: sometimes a request sees data from a previous request's business.

### Pitfall 5: Telegraf Shared Instance State

**What goes wrong:** A single global `Telegraf` instance is used for multiple bot tokens. Middleware, handlers, or context from bot A bleeds into bot B's message processing.

**Why it happens:** Telegraf stores handler functions and context on the instance. Sharing the instance across bots causes race conditions.

**How to avoid:**
- Create one `Telegraf` instance per bot token. Store in a `Map<webhookId, Telegraf>`.
- Retrieve the correct instance by webhook ID before calling `.handleUpdate()`.
- Test with multiple bots in parallel: send messages to two different webhook IDs, verify each routes to the correct handler.

**Warning signs:**
- A single global `bot = new Telegraf(token)` that's used for multiple tokens.
- A test with two webhook IDs fails intermittently (race condition symptom).

### Pitfall 6: Schema Mismatch Between Test and Production Databases

**What goes wrong:** Migration 0003 is applied to the live Neon DB but not the local test DB (or vice versa). Tests pass locally (using old schema) but fail in CI (new schema). Or live app crashes because code expects the new columns.

**Why it happens:** Neon and local PostgreSQL are separate instances. Migrations must be applied to both.

**How to avoid:**
- Apply migration to both simultaneously in a single wave (precedent from Phase 3).
- Run the full test suite after each migration deployment to catch schema mismatches.
- Document the migration execution in PLAN.md's verification steps.

**Warning signs:**
- A test fails in CI but passes locally.
- "column 'webhook_id' does not exist" error on live DB.

### Example 1: Telegraf Webhook Mode with Per-Bot Registry

**Source:** [Telegraf webhook guide](https://telegraf.js.org/) + [RandevuClaw CONTEXT.md](./04-CONTEXT.md)

```typescript
// src/telegram/registry.ts
import { Telegraf } from 'telegraf';
import { logger } from '../utils/logger';

const botRegistry = new Map<string, Telegraf>();

export function getOrCreateBotInstance(webhookId: string, botToken: string): Telegraf {
  if (botRegistry.has(webhookId)) {
    return botRegistry.get(webhookId)!;
  }
  logger.info({ webhookId }, 'Creating new Telegraf instance');
  const bot = new Telegraf(botToken);
  // Do NOT call bot.launch() in webhook mode — the handler calls .handleUpdate()
  botRegistry.set(webhookId, bot);
  return bot;
}

export function clearBotRegistry(): void {
  botRegistry.clear();
}

export function getBotInstance(webhookId: string): Telegraf | undefined {
  return botRegistry.get(webhookId);
}
```

### Example 2: Webhook Handler with Per-Business Routing and RLS Context

**Source:** [RandevuClaw refactored webhook handler](./04-CONTEXT.md#D-04-Route-is-webhooks-telegram-webhookId)

```typescript
// src/webhooks/telegram.ts (excerpt)
import express, { Router, Request, Response } from 'express';
import crypto from 'crypto';
import { getOrCreateBotInstance } from '../telegram/registry';
import { findBusinessByWebhookId } from '../database/queries';
import { withBusinessContext } from '../database/queries';
import { TelegramUpdate } from '../telegram/types';
import { logger } from '../utils/logger';

async function handleTelegramWebhookPost(req: Request, res: Response): Promise<void> {
  try {
    const headerValue = req.headers['x-telegram-bot-api-secret-token'] as string | undefined;
    const webhookId = req.params.webhookId as string;

    // Step 1: Find business by webhook ID
    const business = await findBusinessByWebhookId(webhookId);
    if (!business || !business.webhookSecret || !business.botToken) {
      logger.warn({ webhookId }, 'Webhook ID not found or incomplete');
      res.status(404).send('Not Found');
      return;
    }

    // Step 2: Verify secret with constant-time comparison
    const headerBuffer = Buffer.from(headerValue ?? '');
    const secretBuffer = Buffer.from(business.webhookSecret);

    try {
      crypto.timingSafeEqual(headerBuffer, secretBuffer);
    } catch {
      logger.warn({ webhookId }, 'Invalid webhook secret');
      res.status(401).send('Unauthorized');
      return;
    }

    // Step 3: Get the correct Telegraf instance and handle update
    const update = req.body as TelegramUpdate;
    const bot = getOrCreateBotInstance(webhookId, business.botToken);

    // Step 4: Wrap all downstream DB operations in RLS context
    await withBusinessContext(business.id, async () => {
      await bot.handleUpdate(update);
    });

    res.status(200).send('OK');
  } catch (err) {
    logger.error({ err }, 'Telegram webhook handler failed');
  } finally {
    if (!res.headersSent) res.status(200).send('OK');
  }
}

const router = Router();
router.post('/:webhookId', express.json(), handleTelegramWebhookPost);
export default router;
```

### Example 4: Transaction Wrapper with RLS Context

**Source:** [Drizzle transaction docs](https://orm.drizzle.team/docs/transactions) + [PostgreSQL SET LOCAL](https://www.postgresql.org/docs/15/sql-set.html)

```typescript
// src/database/queries.ts (excerpt)
import { db } from './client';
import { sql } from 'drizzle-orm';

export async function withBusinessContext<T>(
  businessId: string | number,
  callback: () => Promise<T>
): Promise<T> {
  return db.transaction(async (tx) => {
    // Set RLS context for this transaction only
    await tx.execute(sql`SET LOCAL app.current_business_id = ${String(businessId)}`);

    // All queries executed within this callback are filtered by RLS
    return callback();
  });
}

// Example usage in webhook handler:
export async function handleIncomingMessage(
  businessId: number,
  senderTelegramId: string,
  messageText: string
) {
  return withBusinessContext(businessId, async () => {
    // This query will only return messages for the current business, even if it has no WHERE clause
    const recentMessages = await db.select().from(messagesTable);

    // Insert a new message — RLS policy enforces business_id matches context
    await db
      .insert(messagesTable)
      .values({
        businessId,
        senderPhone: senderTelegramId,
        messageBody: messageText,
        status: 'received',
      });

    return recentMessages;
  });
}
```

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No | (Not applicable — bot authentication is via Telegram API only) |
| V3 Session Management | No | (Webhooks are stateless; no user sessions) |
| V4 Access Control | Yes | RLS policies enforce per-business row access; randevuclaw_app role has no superuser privileges |
| V5 Input Validation | Yes | Webhook secret verification via constant-time `crypto.timingSafeEqual`; Telegraf validates JSON structure |

### Known Threat Patterns for Multi-Tenant Telegram Bot Architecture

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| **Webhook Secret Timing Attack** | Tampering | Use `crypto.timingSafeEqual` for constant-time comparison. Never use `===` or `.equals()`. |
| **Bot Token Exposure in Logs** | Information Disclosure | Redact bot tokens from all logging; log only webhook UUID. Audit log retention per GDPR. |
| **Cross-Tenant Row Access via Missing RLS Policy** | Information Disclosure | Enforce RLS policies at database layer. Defense-in-depth: both application WHERE clause AND RLS policy required. |
| **RLS Context Leakage via Shared Connection** | Information Disclosure | Use `SET LOCAL` (transaction-scoped) context, never `SET` (session-scoped). Test with connection pooling to verify isolation. |
| **Unverified Webhook Signature** | Spoofing | Verify `X-Telegram-Bot-Api-Secret-Token` header before processing any update. Reject with 401 if missing or invalid. |
| **Shared Telegraf Instance State** | Tampering / Elevation of Privilege | One Telegraf instance per bot. Middleware/context from bot A must not affect bot B. Use Map registry to isolate instances. |
| **Webhook ID Collision (UUID)** | Spoofing | UUID-v4 has negligible collision probability. Use cryptographically secure random generation (`crypto.randomBytes`). Monitor for duplicates on insert. |

### Data Protection Notes

- **Bot Tokens & Webhook Secrets:** Stored in PostgreSQL (Neon) as plaintext columns; assume database is trusted. Neon supports encryption-at-rest (pay feature); acceptable for PoC as risk is low (bot tokens are ephemeral; rotation via Phase 5 onboarding).
- **Conversation Data:** All messages stored in `messages`, `conversation_turns` tables with `business_id` RLS filter. GDPR right-to-deletion (Phase 6) will provide deletion mechanism.
- **Audit Trail:** Webhook secret changes (if any) are not currently logged; Phase 6 can add audit log. Acceptable for PoC.
