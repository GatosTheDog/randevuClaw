<objective>
Build the session cancellation notification poller (CLSS-03). When an owner cancels a session via the cancel_session tool, the DB row is marked isCancelled=true synchronously. This poller — running every 6 hours — finds cancelled sessions not yet notified and sends Greek messages to all booked clients. A deduplcation table (sessionCancellationNotifications) prevents duplicate sends on poller re-runs.

Purpose: Broadcast notifications must be async (not in the tool handler) to avoid blocking the owner's chat response on network delays. The poller pattern is proven in Phase 9 (membership-expiry.ts).
Output: src/scheduler/session-cancellation.ts, sessionCancellationNotifications Drizzle table added to schema.ts, migration SQL, setInterval registration in src/index.ts.
</objective>

    Module header comment: "Phase 10: Session Cancellation Notification Poller (CLSS-03). Finds cancelled session instances not yet notified and sends Greek messages to all booked clients. Runs every 6 hours via setInterval in src/index.ts. Pattern: src/scheduler/membership-expiry.ts (nested try/catch, botTokenStore.run, per-item error isolation)."

    Imports: and, eq, isNull, inArray from 'drizzle-orm'; db from '../database/db'; sessionInstances, sessionCatalog, sessionCancellationNotifications, bookings, clientBusinessRelationships from '../database/schema'; listAllBusinessIds from '../database/queries'; sendTelegramMessage, botTokenStore from '../telegram/client'; logger from '../utils/logger'. Import findBusinessById (or equivalent) from '../database/queries' for botToken lookup.

    Export async function pollSessionCancellations(): Promise<number> — returns count of notifications sent.

    Implementation follows the membership-expiry.ts pattern exactly:
    1. const businessIds = await listAllBusinessIds();
    2. let notificationCount = 0;
    3. Outer loop: for (const businessId of businessIds):
       - try/catch wrapping the entire per-business block (catches DB errors for that business without stopping others)
       - const business = await findBusinessById(businessId);
       - if (!business || !business.botToken) { logger.warn({businessId}, 'No bot token for business, skipping session cancellation check'); continue; }
       
    4. Per-business: find cancelled sessions with no notification yet:
       const cancelledInstances = await db.select({instanceId: sessionInstances.id, sessionDate: sessionInstances.sessionDate, sessionTime: sessionInstances.sessionTime, catalogId: sessionInstances.catalogId}).from(sessionInstances).innerJoin(sessionCatalog, eq(sessionInstances.catalogId, sessionCatalog.id)).leftJoin(sessionCancellationNotifications, eq(sessionInstances.id, sessionCancellationNotifications.sessionInstanceId)).where(and(eq(sessionCatalog.businessId, businessId), eq(sessionInstances.isCancelled, true), isNull(sessionCancellationNotifications.id)));
       
    5. For each cancelled instance:
       - Find booked clients: const bookedClients = await db.select({clientPhone: bookings.clientPhone}).from(bookings).innerJoin(clientBusinessRelationships, and(eq(bookings.clientPhone, clientBusinessRelationships.senderPhone), eq(bookings.businessId, clientBusinessRelationships.businessId))).where(and(eq(bookings.sessionInstanceId, cancelled.instanceId), inArray(bookings.bookingStatus, ['confirmed', 'pending_owner_approval'])));
       - If bookedClients.length > 0:
         - Inner try/catch per client (one failure never blocks others)
         - For each client: const msg = 'Η σεζόν σας στις ${cancelled.sessionDate} ${cancelled.sessionTime} ακυρώθηκε. Παρακαλώ επικοινωνήστε μαζί μας για νέο ραντεβού.';
         - await botTokenStore.run(business.botToken, async () => { await sendTelegramMessage(client.clientPhone, msg); });
         - notificationCount += 1;
       - After processing all clients for this instance: insert dedup row (regardless of whether any client notifications were sent — even a session with 0 bookings should be marked as "processed" to avoid re-querying it every 6 hours):
         - await db.insert(sessionCancellationNotifications).values({sessionInstanceId: cancelled.instanceId}).onConflictDoNothing();
       
    6. Log per business: logger.info({businessId, cancelledCount: cancelledInstances.length, notificationCount}, 'session cancellation sweep complete');
    7. Return notificationCount;

    Important guard: botToken must never appear in logger calls. Use logger.info({businessId, method: 'pollSessionCancellations'}, ...) pattern only.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-15 | Denial of Service | Duplicate cancellation notification broadcast | medium | mitigate | sessionCancellationNotifications UNIQUE on sessionInstanceId; onConflictDoNothing on insert; poller queries isNull(notification.id) to skip already-processed sessions |
| T-10-16 | Denial of Service | Telegram rate-limit on large-class cancellation broadcast | medium | mitigate | Per-client sequential send (not Promise.all concurrency); ~100ms natural gap between await calls; PoC class size ≤ 30 clients — within Telegram rate limits; no explicit throttle needed at this scale |
| T-10-17 | Information Disclosure | botToken logged in cancellation poller | critical | mitigate | botTokenStore.run() accepted pattern from membership-expiry; logger.warn/info calls never include botToken; only {businessId, method} logged |
| T-10-18 | Spoofing | Poller sending cancellation to wrong business clients | high | mitigate | WHERE sessionCatalog.businessId = businessId scopes all session queries to correct business; clientBusinessRelationships JOIN further validates client-business association |
| T-10-19 | Tampering | Concurrent poller runs sending duplicate notifications | medium | mitigate | In-process setInterval at 6h cadence; only one Node.js process runs at a time on fly.io single Machine; onConflictDoNothing on notification dedup insert guards against any unlikely overlap |
