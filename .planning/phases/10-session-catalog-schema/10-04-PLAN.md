## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Gemini NLU → executeOwnerTool | Gemini parses owner Greek message and dispatches tool — input comes from LLM, not owner directly |
| owner chat → assign_client_to_session | client_phone is owner-provided (tool arg from Gemini); must not leak cross-business client data |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-10 | Spoofing | Owner identity in session tool dispatch | high | mitigate | executeOwnerTool only called after owner Telegram identity verified by existing webhook handler (ownerTelegramId match in ai-owner-agent.ts caller); session tools inherit this guard |
| T-10-11 | Tampering | Gemini hallucinating wrong service_name or capacity | medium | mitigate | service_name resolved via case-insensitive partial match against svcList from DB (not Gemini-provided ID); capacity validated as integer > 0 in createSessionCatalogWithExpansion |
| T-10-12 | Elevation | assign_client_to_session booking cross-tenant client | high | mitigate | bookSessionInstance requires businessId ownership guard via sessionCatalog FK chain; client_phone is recorded but not validated against clientBusinessRelationships in Phase 10 (owner-only tool; client validation deferred to Phase 11 client-facing path) |
| T-10-13 | Denial of Service | Uncaught Telegram send failure in assign_client case | low | accept | D-11 pattern: sendTelegramMessage failure propagates to top-level try/catch which returns Greek error to Gemini; owner sees the error and can retry |
| T-10-14 | Tampering | Gemini injection via weekdays array | low | mitigate | buildRRuleString filters unknown weekday strings via lookup table; no raw string passed to rrule; invalid daynames silently dropped (empty BYDAY = caught by RRule.parseString error) |
</threat_model>
