--- src/conversation/function-executor.ts ---

Add import at the top: import { insertSlotlessRequest, listSlotlessRequestsForClient, countSlotlessRequestsSinceCheckin } from '../session/slotless-requests';

Add a new Zod schema near the existing schemas (after BookAppointmentArgsSchema):
  const RequestSlotlessBookingArgsSchema = z.object({
    business_id: z.number().int(),
    service_id: z.number().int(),
    requested_session_date: z.string(),
    requested_session_time: z.string(),
  });

In bookAppointmentTool, after the enforcement check block and BEFORE the insertBooking call, add the slotless fork:

First, check availability by calling checkAvailability. The existing bookAppointmentTool does NOT call checkAvailability before inserting — it relies on the DB unique index for slot conflicts. For the slotless fork, availability must be checked first. Restructure as follows:

After the enforcement check (enfResult) and BEFORE insertBooking, call:
  const availability = await checkAvailability(context.business.id, parsed.service_id, parsed.calendar_date);

If availability.available_slots is an empty array AND context.business.slotlessRequestsEnabled is true:
  - Build idempotencyKey: `slotless:${context.business.id}:${context.clientPhone}:${parsed.service_id}:${parsed.calendar_date}:${parsed.calendar_time}`
  - Call insertSlotlessRequest({ businessId: context.business.id, clientPhone: context.clientPhone, requestedSessionDate: parsed.calendar_date, requestedSessionTime: parsed.calendar_time, serviceId: parsed.service_id, idempotencyKey })
  - Return { success: false, slotless_request_submitted: true, message: 'Δεν υπάρχουν διαθέσιμες θέσεις. Το αίτημά σας υποβλήθηκε στον ιδιοκτήτη για έγκριση.' }

If availability.available_slots is empty AND slotlessRequestsEnabled is false, return the existing-style no-slots error unchanged.

If availability.available_slots is NOT empty, fall through to the existing insertBooking logic (no change to the happy path).

Note: The existing bookAppointmentTool did not call checkAvailability — it accepted any date/time and relied on DB conflict detection. Adding the check here only changes behavior when the slot IS available. The DB unique index remains as the final conflict guard for concurrent requests. There is no regression for the normal booking path: when slots are available, the function proceeds exactly as before.

In alertOwnerNewBooking (SLOT-06), after building the `text` variable and BEFORE calling sendTelegramMessageWithKeyboard, add:

  let slotlessNote = '';
  try {
    const slotlessCount = await countSlotlessRequestsSinceCheckin(business.id, booking.clientPhone, booking.calendarDate);
    if (slotlessCount > 0) {
      slotlessNote = `\nΑιτήματα χωρίς θέση: ${slotlessCount}`;
    }
  } catch (err) {
    logger.warn({ err, bookingId: booking.id }, 'SLOT-06 count fetch failed (best-effort)');
  }

Append slotlessNote to the text: `${text}${slotlessNote}`.

The countSlotlessRequestsSinceCheckin call uses booking.calendarDate as the "since" date. This is an approximation: it counts requests from the same date as the current booking. A future phase may use a dedicated check-in timestamp; for now this matches the SLOT-06 requirement's spirit ("since last check-in").
