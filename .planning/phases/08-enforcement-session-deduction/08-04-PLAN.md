<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client message → bookAppointmentTool | Untrusted client phone; enforcement check happens before any booking insert |
| bookAppointmentTool → sendTelegramMessage (owner alert) | Alert is best-effort after DB commit — failure must not surface as booking error |
| cancelAppointmentTool → cancelBookingWithRefund | Expiry check and credit restore are atomic inside DB transaction |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-04-01 | Tampering | bookAppointmentTool enforcement bypass | high | mitigate | checkEnforcementAndGetMembership called unconditionally before ANY booking path; no way to skip it without going through the function |
| T-08-04-02 | Denial of Service | block policy sends refusal then caller retries | medium | mitigate | Refusal message to client is informational; no retry loop in Gemini — function returns { error } which Gemini surfaces as text response |
| T-08-04-03 | Repudiation | owner alert send failure silently drops ENFC-03 alert | medium | accept | Alert failure logged as warning; DB booking already committed so client is not affected; owner can query bookings manually; accepted per RESEARCH.md anti-patterns guidance |
| T-08-04-04 | Tampering | cancelBookingWithRefund race on cancel | high | mitigate | cancelBookingWithRefund is atomic (db.transaction()) — expiry check inside transaction on locked state |
| T-08-SC | Tampering | npm/pip/cargo installs | low | accept | No new packages |
</threat_model>
