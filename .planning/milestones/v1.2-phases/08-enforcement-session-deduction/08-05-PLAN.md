## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owner Telegram message → NLU tool → DB | Only the authenticated business owner should be able to change enforcement_policy; gated by aiOwnerAgent ownerTelegramId check before executeOwnerTool is reached |
| NLU tool args → Zod validation → DB write | Gemini may produce any string for policy; Zod validation ensures only 'allow'/'block'/'flag' reach the DB |

## STRIDE Threat Register

```yaml
threats:
  - threat_id: T-08-10
    title: Non-owner sets enforcement policy via policy spoofing
    asvs_ref: V4.1
    severity: high
    vector: A client (not the business owner) sends a message that triggers set_enforcement_policy; the policy is changed on behalf of a business the sender does not own
    impact: Client disables enforcement ('allow') to bypass membership checks; or blocks all bookings ('block') as sabotage
    disposition: mitigate
    mitigation: "set_enforcement_policy tool is only reachable via executeOwnerTool, which is only called from aiOwnerAgent, which is only invoked when the sender's Telegram ID matches business.ownerTelegramId (existing check in telegram.ts at the aiOwnerAgent dispatch point). No new owner auth needed."

  - threat_id: T-08-11
    title: NLU injection of invalid policy value (Gemini hallucinates a non-enum policy)
    asvs_ref: V5.3
    severity: medium
    vector: Gemini returns a policy value outside the enum (e.g., 'strict', 'deny') in the function call response
    impact: setBusinessEnforcementPolicy called with an invalid value; DB CHECK constraint would catch it, but returns a 500-style error to the owner
    disposition: mitigate
    mitigation: "Zod .enum(['allow','block','flag']) in SetEnforcementPolicySchema validates args before any DB call. Invalid value returns Greek error string to owner without touching DB. DB CHECK constraint is a secondary guard (defense in depth per RESEARCH.md open question 1)."

  - threat_id: T-08-12
    title: RLS bypass on enforcement_policy UPDATE (set_enforcement_policy without withBusinessContext)
    asvs_ref: V4.3
    severity: high
    vector: executeOwnerTool case calls handleSetEnforcementPolicy without wrapping in withBusinessContext; setBusinessEnforcementPolicy runs via admin db user (bypasses RLS)
    impact: A single compromised business.id input could update another business's policy; no tenant isolation
    disposition: mitigate
    mitigation: "executeOwnerTool 'set_enforcement_policy' case wraps the call in withBusinessContext(business.id, ...), exactly as the 'list_packages' case does. This sets the RLS context to the authenticated business before the UPDATE executes."
```
