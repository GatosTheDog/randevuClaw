<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| withBusinessContext transaction → getConn() calls | All Phase 8 DB writes must flow through getConn() to share the appDb.transaction(); using db.transaction() directly opens a separate connection and breaks atomicity |
| concurrent webhook deliveries → membership row | Two simultaneous booking requests for the same client hit the same membership row; SELECT FOR UPDATE serializes access |

## STRIDE Threat Register

```yaml
threats:
  - threat_id: T-08-01
    title: Concurrent session deduction race (overselling sessions)
    asvs_ref: V4.1
    severity: high
    vector: Two simultaneous Telegram webhook deliveries for the same client both read sessionsRemaining=1 before either decrements; both proceed to book and deduct
    impact: Client books two sessions with only one remaining; membership balance goes negative; business loses revenue
    disposition: mitigate
    mitigation: "getActiveMembershipForDeduction appends .for('update') (drizzle-orm 0.45.2 pg-core, verified in RESEARCH.md) — acquires a row-level PostgreSQL lock inside the withBusinessContext appDb.transaction(). Second concurrent transaction blocks until first commits, then re-reads the updated sessionsRemaining."

  - threat_id: T-08-02
    title: Double session deduction via Telegram message redelivery
    asvs_ref: V4.2
    severity: high
    vector: Telegram's retry mechanism redelivers the same booking update (e.g., server returned 5xx); bookAppointmentTool and deductSession run again with the same bookingId
    impact: sessionsRemaining decremented twice for a single booking; incorrect balance reported to client
    disposition: mitigate
    mitigation: "deductSession uses idempotency_key='booking:<bookingId>:deduction' with onConflictDoNothing() + returning() check. Second call finds inserted.length=0 and returns before the counter UPDATE — no duplicate decrement possible."

  - threat_id: T-08-09
    title: Information disclosure via membership row accessible across tenants
    asvs_ref: V4.3
    severity: medium
    vector: getActiveMembershipForDeduction or findMembershipByBooking called without proper business scoping
    impact: Membership data from one business visible to another tenant
    disposition: mitigate
    mitigation: "getActiveMembershipForDeduction includes businessId in WHERE clause. All getConn() calls inside withBusinessContext are scoped by PostgreSQL RLS to the active business tenant. findMembershipByBooking relies on bookingId which is already RLS-scoped at insert time."
```
