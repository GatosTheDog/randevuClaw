<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram message → webhooks/telegram.ts | senderTelegramId comes from Telegram webhook; billing operations must validate it is the business owner before any mutation |
| Gemini tool call → executeOwnerTool | tool name and args come from Gemini; args must be Zod-validated in tools.ts before DB write |
| callback_data → parseCallbackData | callback_data is user-supplied string; regex must validate format before passing parsed IDs to handlers |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-01 | Spoofing | Telegram callback_query ownership in handleCallbackQuery | high | mitigate | handleCallbackQuery resolves businessId via findBusinessByOwnerTelegramId(senderTelegramId); all billing handlers receive and validate senderTelegramId against owner_telegram_id; mismatch logs warn and returns without action |
| T-07-02 | Tampering | Gemini NLU injection via package name | high | mitigate | executeOwnerTool create_package case calls handleCreatePackage which uses CreatePackageSchema.safeParse — malformed args from Gemini return Greek error, no DB write; Drizzle parameterized queries in billing/queries.ts prevent SQL injection |
| T-07-03 | Elevation | Unauthorized billing ops | high | mitigate | OWNER_TOOLS are only used in the aiOwnerAgent path which is gated to onboarding-done businesses (see existing routing in telegram.ts); billing callbacks validated against owner identity |
| T-07-04 | Tampering | Double membership on webhook replay | high | mitigate | Handled at DB layer (07-02: idempotencyKey UNIQUE on membership_ledger); payment-flow.ts handleConfirmMembership calls createMembership which enforces this; duplicate callback re-plays fail silently at the unique constraint level |
| T-07-05 | Tampering | Price tampering in callback_data | medium | mitigate | parseCallbackData extracts only integer IDs from billing callbacks; price_cents is never encoded in callback_data (enforced in payment-flow.ts showPackageSelection — verified in 07-04 acceptance criteria); handleConfirmMembership fetches fresh price from billingPackages table by packageId |
| T-07-06 | Information Disclosure | Multi-tenant data leak | high | mitigate | All billing DB operations in payment-flow.ts handlers use withBusinessContext; businessId is derived from authenticated senderTelegramId via findBusinessByOwnerTelegramId, not from untrusted callback_data; callback_data IDs are re-validated against DB ownership |
</threat_model>
