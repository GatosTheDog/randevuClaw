## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Gemini tool call → tools.ts handler | args is Record<string, unknown> from Gemini — must be Zod-validated before any DB write |
| Telegram callback_query → payment-flow.ts | callback_data is untrusted string from Telegram client — must validate sender ownership before any DB mutation |
| payment-flow.ts → billing/queries.ts | Must wrap all DB calls in withBusinessContext to activate RLS |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-01 | Spoofing | callback_query ownership in payment-flow.ts | high | mitigate | handleConfirmMembership, handleCancelPackage, handleConfirmPackage all validate senderTelegramId against findBusinessByOwnerTelegramId before any mutation; log warn and return on mismatch |
| T-07-02 | Tampering | Gemini NLU injection via package name | high | mitigate | handleCreatePackage uses CreatePackageSchema.safeParse(args) — all 4 fields validated via Zod before createPackage() is called; malformed args return Greek error string, no DB write |
| T-07-03 | Elevation | Unauthorized billing ops | high | mitigate | All DB calls in payment-flow.ts wrapped in withBusinessContext(businessId, ...); businessId validated against senderTelegramId in each handler |
| T-07-04 | Tampering | Double membership on webhook replay | high | mitigate | Handled at DB layer (07-02 schema idempotencyKey UNIQUE + 07-03 createMembership transaction) — payment-flow.ts trusts DB layer idempotency |
| T-07-05 | Tampering | Price tampering in callback_data | medium | mitigate | showPackageSelection encodes only package IDs in callback_data ("billing:package:{clientRelId}:{packageId}"); price_cents is in button text label only; handleConfirmMembership fetches price fresh from DB on confirmation |
| T-07-06 | Information Disclosure | Multi-tenant data leak | high | mitigate | All read queries in payment-flow.ts and tools.ts use withBusinessContext; businessId is always passed explicitly and validated against senderTelegramId |
</threat_model>
