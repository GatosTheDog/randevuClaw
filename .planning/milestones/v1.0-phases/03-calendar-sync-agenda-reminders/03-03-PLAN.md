## Phase Goal

**As a** business owner using RandevuClaw, **I want to** have every confirmed booking automatically reflected on my Google Calendar, get a daily Telegram summary of today's appointments, and have my clients automatically reminded before their visit, **so that** I never manually update a calendar or chase a no-show client, and my clients show up on time without me lifting a finger.

> Note: see Plan 03-01's identical note on ROADMAP.md's Phase 3 Goal-line format. This plan is the one human-only step OWNR-04 needs to be observable with real data: Plan 03-02 built the OAuth flow and Calendar sync code, but only a human can click "Allow" on Google's real consent screen (D-05/D-07 -- Claude cannot script another party's account authorization).

<objective>
Run the OAuth consent-flow CLI tool Plan 03-02 built, once per fixture business, so both `pilates-athens` and `hair-salon-athens` end this checkpoint with a real, working `googleRefreshToken` stored in their `businesses` row -- the prerequisite for Plan 03-02's Calendar sync code (already built and unit-tested) to actually write real events to a real calendar, and for the phase's overall goal to be demonstrable end-to-end rather than only unit-test-verified.

Purpose: There is no CLI/API path for a third party's Google account to grant OAuth consent on its own behalf -- this is Google's own security model, not a gap in this project's tooling. Per D-05/D-07, the project owner runs this once per fixture business now (fast, low-friction) rather than deferring it, so Plan 03-02's already-built code has real data to exercise before the phase closes out.
Output: Two fixture businesses, each with a live, working Google Calendar connection.
</objective>
