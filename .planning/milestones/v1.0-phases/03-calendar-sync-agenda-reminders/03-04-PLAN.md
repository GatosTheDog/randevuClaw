## Phase Goal

**As a** business owner using RandevuClaw, **I want to** have every confirmed booking automatically reflected on my Google Calendar, get a daily Telegram summary of today's appointments, and have my clients automatically reminded before their visit, **so that** I never manually update a calendar or chase a no-show client, and my clients show up on time without me lifting a finger.

> Note: see Plan 03-01's identical note on ROADMAP.md's Phase 3 Goal-line format. This plan delivers the second "I want to" clause: OWNR-03, the daily agenda.

<objective>
Implement the 8am-Athens daily agenda: an in-process poller (no cron/Redis, matching the locked stack and Plan 02-05's `expiry-poller.ts` precedent) that, once per business per Athens calendar day, composes a Greek-language summary of that day's confirmed appointments and sends it to the business owner via Telegram.

Purpose: This is the smallest and lowest-risk of Phase 3's three features -- no external API, no new schema, no new credentials. Its entire risk surface is getting the idempotency guard and the DST-safe day-boundary check right (RESEARCH.md Pitfall 5), both of which Plan 03-01 already solved at the query layer via `claimAgendaSlot`'s atomic UPDATE...WHERE...RETURNING. Per D-03, this plan adds no new Telegram send helper -- it calls the existing `sendTelegramMessage` as-is. Per D-04, the sweep's scheduling/business logic (which businesses, which bookings, message composition) stays entirely separate from the Telegram send call, so a future WhatsApp adapter can slot in as a drop-in replacement for that one call site. Per D-12, the poller runs in the SAME process as the Express server and the other Phase 3 pollers -- no `fly.toml` `[processes]` changes.
Output: A working daily agenda, self-contained in one new file plus one new line in `server.ts`.
</objective>
