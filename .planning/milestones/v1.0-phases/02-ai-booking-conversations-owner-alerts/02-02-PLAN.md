## Phase Goal

**As a** client of a Greek service business, **I want to** check availability, book, cancel, reschedule, and ask questions about my appointment entirely through natural Greek conversation on Telegram, **so that** I can manage my booking with zero friction while the owner is reliably alerted to act on it and no two clients ever end up double-booked into the same slot.

> Note: ROADMAP.md's Phase 2 `**Goal**:` line is descriptive prose, not the strict "As a / I want to / so that" format. This story is derived directly from that Goal plus Success Criteria 1-5, reframed for Telegram per CONTEXT.md D-01/D-04, without adding new scope. This plan delivers the channel-transport slice of the story: a client can reach a resolved, consented business over Telegram. The AI booking conversation itself (the "check availability, book, cancel, reschedule, ask questions" verbs) is layered on top starting at Plan 02-04, exactly as Phase 1's Plan 01-02 delivered "reach the right business" before Plan 01-03 layered dedup/consent on top.

<objective>
Stand up the Telegram channel adapter: an outbound Bot API client (send message, send with inline keyboard, answer callback query, edit reply markup — all four primitives every later plan needs) and an inbound webhook handler that verifies Telegram's secret token, deduplicates by `update_id`, resolves the business from the message text by reusing Phase 1's `resolver.ts` and `consent/checker.ts` UNCHANGED (per CONTEXT.md D-03's channel-agnostic-core mandate), and replies in Greek. This proves the full Telegram round trip end-to-end before any AI/booking logic exists, mirroring Phase 1 Plan 01-02's "prove the stack first" approach.

Purpose: Every later Phase 2 plan (availability, AI agent, owner approval, cancel/reschedule/FAQ) sends and receives Telegram messages through this client and this webhook entry point — get the primitives and the dedup/resolution wiring right once, here.
Output: A real (or documented locally-runnable) webhook at `/webhooks/telegram` that resolves business identity from a live Telegram message and replies in Greek, deployed alongside the existing `/webhooks/whatsapp` route.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram (internet) -> Express webhook | Fully untrusted inbound HTTP; every request must be secret-token-verified before any processing |
| Webhook -> Telegram Bot API | Outbound HTTPS call carrying the bot token embedded in the URL path |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-05 | Spoofing | src/webhooks/telegram.ts | critical | mitigate | `X-Telegram-Bot-Api-Secret-Token` compared via direct string equality (Telegram's own documented mechanism — no HMAC exists for Telegram webhooks); unmatched requests get 403 before any body parsing or business logic |
| T-02-06 | Denial of Service | src/webhooks/telegram.ts | medium | mitigate | try/catch/finally always resolves 200 to Telegram regardless of internal errors (matches Phase 1's invariant), preventing Telegram's own retry storm on a slow/erroring handler; malformed `update.message.text` (missing/undefined) defaults to empty string rather than throwing |
| T-02-07 | Information Disclosure | src/telegram/client.ts | medium | mitigate | `TELEGRAM_BOT_TOKEN` sent only as a URL path segment over HTTPS to api.telegram.org, never logged (already covered by Task 1 of Plan 02-01's logger redact list), never surfaced in any error returned to the client |
| T-02-08 | Repudiation / Tampering | src/database/queries.ts insertOrIgnoreTelegramUpdate | medium | mitigate | Every inbound Telegram update (message or callback_query) is durably deduped by `update_id` before any side effect, giving an audit trail and preventing Telegram's own delivery retries from producing duplicate replies |

</threat_model>
