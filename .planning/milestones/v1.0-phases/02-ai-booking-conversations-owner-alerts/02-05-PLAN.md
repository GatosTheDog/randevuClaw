## Phase Goal

**As a** client of a Greek service business, **I want to** check availability, book, cancel, reschedule, and ask questions about my appointment entirely through natural Greek conversation on Telegram, **so that** I can manage my booking with zero friction while the owner is reliably alerted to act on it and no two clients ever end up double-booked into the same slot.

> Note: ROADMAP.md's Phase 2 `**Goal**:` line is descriptive prose, reframed for Telegram per CONTEXT.md D-01/D-04 (see Plan 02-01's identical note for the full derivation). This plan closes the loop Plan 02-04 opened: OWNR-02's "owner receives alert AND can accept or reject" only becomes fully true once the owner's tap is actually processed, and D-09's 2-hour auto-expiry only becomes true once something proactively sweeps and notifies. This is the last plan required for Phase 2's 5 ROADMAP success criteria to all hold.

<objective>
Implement the owner's half of the approval loop: parse and validate `callback_query` data (replacing Plan 02-02's `// TODO Plan 02-05` stub), verify the tapping Telegram user is genuinely the alerted business's owner, apply the approve/reject transition (including the reschedule cascade — confirming the new booking and cancelling the original in one operation), clear the alert's buttons, and notify the client. Separately, implement a lightweight in-process poller (no cron/Redis infrastructure, per the locked stack) that sweeps all businesses' stale pending bookings past their 2-hour `expiresAt` and proactively tells the affected clients, per D-09.

Purpose: Every guardrail AI-SPEC Section 6 assigns to the owner-approval surface (booking_id validation, no-default-to-most-recent) lives here. Getting this right is what makes OWNR-02 and the ROADMAP's 5th success criterion ("owner can accept/reject... no two clients ever end up double-booked") actually true end-to-end, not just true for the booking-creation half already covered by Plan 02-04. This is also where D-08 (owner responds via Telegram inline keyboard buttons, never text commands) is fully realized — Plan 02-04 sent the buttons, this plan is what makes tapping them do something.
Output: A fully closed booking lifecycle — pending -> confirmed/rejected (owner-driven) or pending -> expired (time-driven) — with the client and owner both correctly informed at every transition.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram callback_query -&gt; src/webhooks/telegram.ts | The tapping user's identity (`callback_query.from.id`) is attacker-influenceable in the sense that ANY Telegram user could construct a request with an arbitrary bot-known booking id if they ever saw one; the server must independently verify the tapper IS the alerted business's owner, never trust the button's mere existence as authorization |
| src/conversation/expiry-poller.ts -&gt; Telegram Bot API | Proactive outbound messages triggered by a timer, not a request — must not leak booking details to the wrong chat_id (uses `booking.clientPhone`/`business.ownerTelegramId` sourced directly from the DB row just transitioned, never from any external input) |
