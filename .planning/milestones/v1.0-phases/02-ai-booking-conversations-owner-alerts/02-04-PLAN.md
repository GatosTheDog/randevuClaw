## Phase Goal

**As a** client of a Greek service business, **I want to** check availability, book, cancel, reschedule, and ask questions about my appointment entirely through natural Greek conversation on Telegram, **so that** I can manage my booking with zero friction while the owner is reliably alerted to act on it and no two clients ever end up double-booked into the same slot.

> Note: ROADMAP.md's Phase 2 `**Goal**:` line is descriptive prose, reframed for Telegram per CONTEXT.md D-01/D-04 (see Plan 02-01's identical note for the full derivation). This plan is the load-bearing vertical slice: it turns the channel round-trip (Plan 02-02) and the availability/temporal engine (Plan 02-03) into an actual conversational booking agent. After this plan, a client can hold a real Greek conversation on Telegram and book, cancel, reschedule, or ask a question — the one missing piece is the owner's side of the approval loop (callback_query handling, Plan 02-05), which is why OWNR-02 is only partially delivered here (the alert is sent; acting on it is Plan 02-05).

<objective>
Build the actual AI booking conversation: (1) `aiBookingAgent`, the sequential Gemini function-calling loop with a Greek system prompt grounded in the business's real services/hours (no hallucination) and 429 backoff; (2) `function-executor.ts`, the tool layer implementing check_availability/book_appointment/cancel_appointment/reschedule_appointment with the AI-SPEC Section 6 guardrails (cross-tenant business_id check, booking-ownership check, idempotent-retry disambiguation) baked in; (3) `router.ts`, the channel-agnostic conversation core (D-03) tying consent, Greek-temporal preprocessing, and turn persistence together; and wiring all of it into the Telegram webhook's message branch in place of Plan 02-02's placeholder reply.

Purpose: This is where the phase's five ROADMAP success criteria actually become true. Every guardrail in AI-SPEC.md Section 6 that prevents double-booking, duplicate owner alerts, cross-tenant leakage, and wrong-client cancellation is implemented in this plan's function-executor.ts — get these right once here so Plan 02-05's owner-approval flow only has to consume correctly-shaped booking rows, never re-derive correctness. Telegram is the sole channel this plan targets — WhatsApp is shelved (D-01/D-04) and Viber was ruled out entirely (D-02, no self-service bot signup available), so there is no channel-selection branching to build here.
Output: A live Telegram conversation that books, cancels, reschedules, and answers questions in Greek, with new-booking/reschedule owner alerts already flowing (approval handling itself is Plan 02-05).
</objective>

|----------|-------------|
| Client Telegram message -&gt; Gemini `input` | Fully untrusted natural-language text reaches the LLM; the LLM's own tool-call arguments (including any `business_id` it echoes back) must never be trusted as authorization |
| Gemini tool-call arguments -&gt; function-executor.ts | The LLM is a semi-trusted intermediary: it can be prompt-injected by client text into requesting an action against the wrong business or a booking it doesn't own; every mutation must be re-validated server-side |
| function-executor.ts -&gt; Telegram Bot API (owner alert) | Outbound message carrying business-name/service/date/time/client-phone to the owner's chat; only sent after a real DB row exists, never speculatively |
