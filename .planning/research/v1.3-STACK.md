## Executive Summary

**Existing validated stack (DO NOT change):**
- Node.js 20+ / TypeScript
- Neon PostgreSQL (serverless, free tier 100 CU-hrs/month, doubled Oct 2025)
- Drizzle ORM 0.45.2 (7.4 KB minified, zero runtime dependencies, RLS-ready)
- Telegraf 4.16.3 (per-bot webhook routing, callback_query support verified in v1.2)
- Google Gemini @google/genai 2.10.0+ (function-calling, free tier 15 RPM / 1K RPD)
- fly.io + Supercronic (in-process pollers, no external job queue for PoC)

**Stack additions for v1.3 (minimal):**
1. **`rrule` v2.8.1** — RFC 5545 recurrence rule parsing for recurring session patterns (CLSS). Battle-tested library, 791K weekly npm downloads, native TypeScript types.
2. **Zero new date libraries** — Reuse existing `src/utils/timezone.ts` (Intl API, DST-safe, zero-dependency) for cutoff window logic (CANC) and renewal threshold checks (RENW).
3. **Drizzle schema extensions** — New tables (sessionCatalog, sessionInstances, slotlessRequests) using parent-child FK relationships; no new ORM library.
4. **In-process state machine** — Existing Telegraf callback_query handler for approval buttons; no new messaging library.
5. **7 new business config columns** — Optional per-business flags (booking_mode, cancellation_cutoff_enabled/hours, last_session_threshold_enabled/count, slotless_requests_enabled, allow_multi_booking).

**Migration strategy:**
- ADD schema changes (backward-compatible; all new columns nullable or opt-in).
- NO breaking changes to existing booking flow, enforcement, calendar sync, or reminders.
- All new features disabled by default.

## Recommended Stack Additions

### 1. Session Catalog: Recurring Patterns (CLSS)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| **rrule** | 2.8.1 | Parse and expand RFC 5545 recurrence rules into discrete session instances | RFC 5545 standard (iCalendar format). Native TypeScript types. 791K weekly downloads — battle-tested in Calendly, Google Calendar, Apple Calendar. Handles FREQ, BYDAY, UNTIL, COUNT, INTERVAL. No timezone bugs (expands to local dates). |

**Integration pattern:**

Owner defines: "Monday 09:00 Pilates, capacity 10, recurring every week until Dec 31"
→ Gemini parses Greek input
→ `createRecurringSessionTool(serviceId, capacity, rruleString, dtstart, dtend)`
→ Backend: `rrule.between(dtstart, dtend)` expands to discrete dates
→ CREATE sessionInstance rows per expansion (each with availableSlots = capacity)
→ Client books → deduct from sessionInstance.availableSlots (not shared DB booking slots)

**Why this library:**
- `rrule` standard de facto for JS recurrence; implemented in 1000+ production apps.
- Alternatives like `simple-rrule` exist but have weaker timezone support; `rrule` is battle-tested.
- **Not** using `luxon` / `date-fns` / `temporal`: General-purpose date libs, not RRule-specific; overkill for a single rrule parse step.
- **Not** manual recurrence parsing: Fragile (edge cases: leap years, month-end rollovers, DST), hard to extend.

**Bundle size:** ~28 KB minified (no gzip needed server-side); 0 KB added to Telegram client (server-side parsing only).

### 2. Cutoff Window & Renewal Thresholds (CANC, RENW)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| **Existing timezone.ts** | (0 KB added) | DST-safe date comparisons: "now < (sessionTime - cutoffHours)?" and "lastSession <= thresholdCount?" | Reuse isoDateInAthens() + addCalendarDays() + weekdayOfIsoDate(). Proven in v1.2 billing (6+ months production). Intl API (browser/Node.js standard). Zero dependencies. |

**Integration:**

- New optional column: `businesses.cancellation_cutoff_hours` (DEFAULT NULL = "cancel anytime" mode, backward-compatible).
- On `cancelBookingTool`: Check `isoDateInAthens(now) + cutoffHours < sessionTime`?
  - If YES and cutoff enabled → Show Greek confirmation dialog ("Λήγει η δυνατότητα ακύρωσης σε 2 ώρες. Σίγουρα;").
  - If confirmed → Deduct credit; if cancelled → Restore credit (reuse v1.2 ledger logic).
- New column: `businesses.last_session_threshold_enabled / _count` (DEFAULT false, NULL).
- Renewal sweep: Check `sessionsRemaining <= threshold_count` → Send 1× per membership nudge ("Μόνο 3 σεάνς απομένουν...").

**Why no new library:**
- `date-fns-tz` would add 15+ KB gzipped for timezone-aware arithmetic; existing manual approach proven, smaller.
- Intl API is stable, timezone-aware, DST-safe, no DST bugs.
- Cutoff logic: Just integer comparison + Intl date formatting. No library needed.

### 3. Slotless Booking Requests (SLOT)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| **Existing Telegraf + Drizzle** | (0 KB added) | Request queue state machine (pending → approved/rejected) stored in DB, callback_query inline buttons for approval | Telegraf already handles callback_query in v1.2 (owner-approval flow). Drizzle atomic transactions prevent concurrent state conflicts. No new library needed. |

**Integration:**

- New table: `slotlessRequests (id, businessId, clientPhone, requestedDate, requestedTime, reason, requestStatus, ownerResponseAt)`.
- Client: "Θέλω να κλείσω Παρασκευή απόγευμα" (no available slots) → Gemini → `createSlotlessRequestTool` → INSERT pending request.
- Owner alert: "📅 Παράκληση: [ClientName] Παρασκευή 17:00" with inline keyboard: `[✅ Approved] [❌ Rejected]`.
- Button press → callback_query → `UPDATE slotlessRequests SET requestStatus = 'approved'` (atomic, RLS-checked).
- On approval: `insertBooking()` with same booking flow (calendar sync, enforcement, reminders).

**Why no external queue library:**
- `Bull` / `BullMQ`: Redis dependency (not in free tier; costs $$ for managed Redis).
- `Temporal.io`: Enterprise-grade orchestration; overkill for 1-business PoC.
- `Agenda`: Requires MongoDB (breaks single-DB principle; adds cost).
- **DB-backed state machine:** Simpler, persistent, auditable, requires only Drizzle (already in stack).

## Schema Extensions: New Tables & Columns

### New Table: sessionCatalog

Represents a recurring session definition (owner-created template).

```sql
CREATE TABLE session_catalog (
  id SERIAL PRIMARY KEY,
  business_id INTEGER NOT NULL REFERENCES businesses(id),
  service_id INTEGER NOT NULL REFERENCES services(id),
  capacity_count INTEGER NOT NULL, -- Max concurrent clients per session
  rrule_string TEXT, -- RFC 5545, e.g. 'FREQ=WEEKLY;BYDAY=MO,WE,FR'
  dtstart TEXT NOT NULL, -- "YYYY-MM-DD" session start date (Athens local)
  dtend TEXT, -- nullable, recurrence end date (Athens local)
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
-- UNIQUE INDEX: (business_id, service_id, dtstart, rrule_string)
-- Prevents duplicate recurrence definitions per business/service.
```

**Drizzle relation:** Parent of sessionInstances (one catalog → many instances).

### New Table: sessionInstances

Represents a discrete session date/time (result of rrule expansion).

```sql
CREATE TABLE session_instances (
  id SERIAL PRIMARY KEY,
  session_catalog_id INTEGER NOT NULL REFERENCES session_catalog(id),
  calendar_date TEXT NOT NULL, -- "YYYY-MM-DD"
  calendar_time TEXT NOT NULL, -- "HH:MM"
  available_slots INTEGER NOT NULL, -- Remaining capacity
  is_cancelled BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
-- PARTIAL UNIQUE INDEX: (session_catalog_id, calendar_date, calendar_time) WHERE is_cancelled = false
-- Allows re-cancellation and re-booking after cancellation.
```

**Drizzle relation:** Child of sessionCatalog. On booking, deduct available_slots atomically (SELECT FOR UPDATE, same pattern as v1.2 membership deduction).

### New Table: slotlessRequests

Represents a client's booking request when no open slot exists (approval queue).

```sql
CREATE TABLE slotless_requests (
  id SERIAL PRIMARY KEY,
  business_id INTEGER NOT NULL REFERENCES businesses(id),
  client_phone TEXT NOT NULL,
  requested_date TEXT NOT NULL, -- "YYYY-MM-DD"
  requested_time TEXT, -- nullable, client may say "sometime Friday"
  reason TEXT, -- client's message/reason (optional)
  request_status TEXT NOT NULL DEFAULT 'pending', -- 'pending' | 'approved' | 'rejected'
  owner_response_at TIMESTAMP, -- when owner approved/rejected
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
-- INDEX: (business_id, request_status) for listing pending requests
-- COMPOSITE UNIQUE (client_phone, requested_date, requested_time) if all non-null
```

**State machine:** pending → approved (convert to booking) or rejected (discard).

### Modified Existing Table: bookings

Add optional reference to session instance (for session-catalog bookings).

```sql
-- Add nullable column:
ALTER TABLE bookings ADD COLUMN session_instance_id INTEGER REFERENCES session_instances(id);
-- Regular slot bookings have NULL; session-instance bookings reference this column.
-- Enables capacity deduction from sessionInstance.available_slots instead of general slot availability.
```

### Modified Existing Table: businesses

Add 7 new optional config columns (all nullable, backward-compatible).

```sql
-- Add columns (all nullable, defaults preserve opt-in behavior):
ALTER TABLE businesses ADD COLUMN booking_mode TEXT DEFAULT 'slots'; -- 'slots' | 'sessions'
ALTER TABLE businesses ADD COLUMN cancellation_cutoff_enabled BOOLEAN DEFAULT false;
ALTER TABLE businesses ADD COLUMN cancellation_cutoff_hours INTEGER; -- nullable
ALTER TABLE businesses ADD COLUMN last_session_threshold_enabled BOOLEAN DEFAULT false;
ALTER TABLE businesses ADD COLUMN last_session_threshold_count INTEGER; -- nullable
ALTER TABLE businesses ADD COLUMN slotless_requests_enabled BOOLEAN DEFAULT false;
ALTER TABLE businesses ADD COLUMN allow_multi_booking BOOLEAN DEFAULT false; -- allow 2+ bookings same day
```

**Why Drizzle relations (not raw SQL)?**
- Parent-child FK chains enable prefetch queries without N+1 overhead.
- RLS policies inherit businessId via catalog → instance chain.
- Type-safe in TypeScript; no string escaping, no SQL injection.

## Risk Mitigation

| Risk | Mitigation | Owner |
|------|-----------|-------|
| **rrule DST edge case** (e.g., Feb/Oct Greece transitions) | Write 5+ tests for DST transitions (follow existing timezone.test.ts pattern). Verify expansion for Feb 29, Oct 25 (DST dates). | Phase 1 (CLSS) |
| **Session capacity race** (2 clients book last slot concurrently) | Reuse SELECT FOR UPDATE pattern from v1.2. Add partial unique index to prevent duplicates. | Phase 2 (session booking) |
| **Slotless queue unbounded growth** | Add cron job to archive old (rejected >30 days) to audit table (defer to Phase 4). Monitor db size. | Phase 4 (SLOT) |
| **Cutoff conflicts with past bookings** | Pre-check existing bookings before allowing new cutoff window. No retroactive changes to past bookings. | Phase 3 (CANC) |
| **Owner forgets new features** | New onboarding steps (26–31) set sensible defaults (disabled). No surprise behavior. Test onboarding flow. | Phase 6 (ONB extensions) |
| **Multi-booking abuse** (client books 100 same-day sessions) | Add per-business rate limit if allow_multi_booking=true. Defer implementation to Phase 2 if needed. | Phase 2 |
