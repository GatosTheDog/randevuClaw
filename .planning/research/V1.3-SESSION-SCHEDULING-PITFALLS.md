# Domain Pitfalls: v1.3 Session Scheduling & Slotless Bookings with Session-Credit Billing

**Domain:** Appointment booking platform (Greek service businesses) with session-credit billing
**Researched:** 2026-07-22
**Milestone:** v1.3 Studio Session Scheduling & Slotless Bookings

**Overall Confidence:** HIGH (pitfalls informed by Phase 8 session-credit patterns, race-condition experience, and timezone infrastructure already proven in production)

---

## Executive Summary

Adding session-catalog booking (fixed session capacity, recurrence, direct assignment), slotless request workflows (approve/reject), and cancellation cutoffs to the existing v1.2 booking system with session-credit billing introduces **six critical pitfall zones**:

1. **Capacity races** — Two concurrent client bookings claim the last spot; both succeed in application layer, second fails in DB
2. **State machine explosion** — Slotless request → approval → booking creation path creates orphaned requests when client membership expires mid-workflow
3. **Timezone bugs under DST** — Cancellation cutoff calculation ("2 hours before session in Athens time") breaks on DST transition boundaries
4. **Booking mode switch orphaning** — Toggling `booking_mode` from open_slots → fixed_sessions mid-operation loses existing bookings that don't fit new schema
5. **Telegram rate limit cascade** — Mass renewal broadcast (> 15 Gemini req/min, > per-chat Telegram limits) throttles all other operations
6. **Recurring session expansion** — Creating 52 future sessions atomically stalls concurrent bookings; lazy expansion races if not protected

All six pitfalls interact with the existing `SELECT FOR UPDATE` session-deduction mechanism (from Phase 8). Prevention requires careful transaction scoping, idempotency keys, and explicit state machine enforcement.

## Critical Pitfalls

### Pitfall 1: Capacity Race on Session Booking (Fixed Sessions Mode)

**What goes wrong:**
Two clients simultaneously request the last open seat in a session. App layer checks `session.capacityRemaining > 0` for both, both see the same remaining count, both proceed to create bookings. First booking succeeds; second fails with `UNIQUE constraint violation`. Client sees an error after a successful intermediate state check.

**Why it happens:**
- Session capacity tracking exists at DB layer (via `sessionCatalog.capacityRemaining` counter) or app layer (derived from `SELECT COUNT(...) FROM bookings WHERE session_id = X`)
- Between the SELECT and INSERT booking, another request acquires the last seat
- The `UNIQUE constraint on (business_id, calendar_date, calendar_time)` from Phase 1 still exists but only blocks **per time slot**, not per session instance
- If sessions are defined as sessions + recurrence, the new schema adds (session_id, recurrence_date) to the slot uniqueness tuple, but the check-then-insert window remains

**Consequences:**
- Unpredictable failures at high concurrency (2+ clients booking within milliseconds)
- Client sees "booking conflict" after availability check passed
- Owner sees partially-accepted bookings (first succeeded, second rejected)
- Session analytics become unreliable (count ≠ actual bookings)

**Prevention:**
1. **Do NOT check capacity in app layer** — move the entire check+insert into a single DB transaction with `SELECT FOR UPDATE` on the session row (exact pattern: use `getConn()` inside `withBusinessContext` transaction, matching Phase 8 pattern from src/billing/queries.ts)
2. **UNIQUE constraint must prevent double-booking per session instance**: Add `UNIQUE INDEX on (session_id, client_phone)` — one booking per client per session instance
3. **Idempotency key must include session_id**: Change from `${requestId}:${call.id}` to `${requestId}:${call.id}:${sessionId || ''}` so retries on the same session don't mistakenly resolve to an old booking from a different session
4. **Test the race explicitly**: Write a test that spawns 3 concurrent booking requests for a session with capacity=1; assert exactly 1 succeeds, 2 fail with `error: 'slot_taken'` or similar; use `Promise.all()` with `Promise.allSettled()` to capture all outcomes

**Detection:**
- Watch for ERROR logs with "Unique constraint violation" + session_id context
- Monitor `bookings.COUNT(WHERE session_id = X)` vs. `sessionCatalog.capacityRemaining` — if COUNT > capacity, the race was not prevented
- Add observability: log the capacity check + INSERT in the same span so retries are visible

### Pitfall 2: Slotless Request Orphaning on Membership Expiry

**What goes wrong:**
1. Client sends "I want to book Sunday, no open slots" → `slotless_requests` row created with status='pending', linked to client + service
2. Owner opens the approval UI, sees the request, taps "Approve"
3. App creates a booking row (status='pending_owner_approval') + links it to the request (status='approved')
4. Meanwhile, the client's membership expires (cron sweep runs, or manual cancellation)
5. Owner tries to confirm the booking (owner taps "Accept" on the booking alert)
6. The booking.create succeeded **before** membership expiry, so no enforcement was applied
7. Client books the appointment despite no active membership, defeating the enforcement policy

**Alternative scenario:**
1. Slotless request created
2. Owner taps "Approve" → intermediate `slotless_requests.status='approved'` row written
3. Network hiccup before the booking INSERT completes
4. Retry logic re-runs the approval, but the idempotency key on the booking INSERT catches the conflict
5. New booking row is created from a **second approval** because the request state machine allows re-approval
6. Client is charged/deducted twice for the same approval

**Why it happens:**
- Slotless workflow introduces a new asynchronous approval step before booking creation, breaking the "check membership → create booking atomically" pattern that worked in open_slots mode
- `slotless_requests` table has no FK to `memberships`, so the request persists even after the membership expires
- The approval callback (owner taps "Approve") can succeed even if membership became invalid **between** request creation and approval
- No idempotency guard on the "approve" step itself — owner re-tapping "Approve" can create multiple bookings

**Consequences:**
- Expired clients book appointments despite enforcement policy
- Session credits deducted from non-existent or expired memberships
- Owner sees stale requests for clients who are no longer members
- Billing audit reveals phantom deductions

**Prevention:**
1. **Re-check membership validity at approval time** (not just at request time): Use `getActiveMembershipForDeduction()` with `SELECT FOR UPDATE` inside the approval transaction
2. **Approval transaction must be atomic**: membership check + request status update + booking creation all in one `withBusinessContext` transaction
3. **Request idempotency must block re-approval**: Add `UNIQUE INDEX on (slotless_request_id) WHERE status IN ('pending', 'approved')`
4. **Explicit request lifecycle**: 'pending' → 'approved' → 'booking_created' (success) or 'pending' → 'rejected' (owner decline); add `rejectedAt` timestamp
5. **Track booking←request association**: Add `slotless_request_id` FK to bookings table; query `SELECT * FROM bookings WHERE slotless_request_id = $1` to verify exactly 1 booking per request
6. **Stale request cleanup**: Run periodic sweep (same as membershipExpiry sweep in Phase 9) that marks requests as 'expired' if status='pending' AND createdAt < 48h ago, or status='approved' AND booking creation failed + retries exhausted

**Detection:**
- Query: `SELECT COUNT(*) FROM bookings WHERE slotless_request_id IN (SELECT id FROM slotless_requests WHERE status != 'booking_created')` — if result > 0, state machine is broken
- Log approval callback entry + membership check + booking INSERT in same trace
- Alert if a booking is created for an expired membership (compare booking.createdAt vs membership.expiresAt)

### Pitfall 3: Cancellation Cutoff Timezone Bugs (DST Boundary Failures)

**What goes wrong:**
Business sets `cancellation_cutoff_hours = 2`. Client books Sunday 2026-10-25 09:00 Athens time (during DST transition day: clocks go back 1 hour at 04:00 that morning, UTC+3 → UTC+2). Client wants to cancel at 08:00 on the 25th. The cutoff check should say "cancel allowed — only 1 hour until appointment." Instead:

1. Cutoff calculation uses hardcoded `new Date().getTime() + 2 * 3600 * 1000` (add 2 hours in UTC)
2. Or: uses `new Date(bookingDateTime).getTime() - 2 * 3600 * 1000` (subtract 2 hours)
3. Either way, the "2 hours" offset is applied in UTC time, not Athens wall-clock time
4. On the DST boundary, UTC and Athens time have different offset ratios
5. Cancellation is rejected when it should be allowed (or vice versa)

**Example timeline:**
- Booking: Sunday 2026-10-25 09:00 Athens local time (during DST: UTC+3)
- 02:59:59 UTC = 05:59:59 Athens UTC+3 (last minute before clock-back)
- 03:00:00 UTC = 04:00:00 Athens UTC+2 (clocks went back; now UTC+2 for the rest of the day)
- 07:45 Athens UTC+2 = 05:45 UTC (1h 15m before the 09:00 UTC+2 = 07:00 UTC appointment)
- Cancellation at 07:45 should be allowed (< 2h cutoff)
- But naive UTC arithmetic may compute the difference incorrectly because the offset changed

**Why it happens:**
- The existing `athensEndOfDay()` utility correctly handles DST by reading the offset at a specific instant (already in src/utils/timezone.ts)
- But the new cancellation cutoff logic may reuse naive millisecond arithmetic instead of calling DST-safe utilities
- Not all developers recognize DST transitions as a special case; they assume UTC timestamps always add/subtract correctly
- Test coverage often doesn't include DST boundary dates (October 25, March 31 for Europe/Athens)

**Consequences:**
- Clients randomly denied cancellations that should be allowed
- Owner receives complaints but cannot reproduce (off-by-one only on DST days)
- Credits are mistakenly forfeited or incorrectly allowed
- Opposite risk: cancellations allowed when they should be blocked, owner loses revenue

**Prevention:**
1. **Use the existing DST-safe utilities** (`isoDateInAthens` + `addCalendarDays` + `athensEndOfDay`) for all time calculations, never naive UTC offsets
2. **Cutoff logic must use wall-clock arithmetic in Athens timezone**, not UTC offsets: compute cutoff date via `addCalendarDays(bookingDate, -1)` if hours < cutoffHours, then compare wall-clock times
3. **Test explicitly on DST boundaries** (2026-10-25, 2027-03-28): create bookings 1h, 2h, 3h before/after the transition, verify cutoff decisions are consistent
4. **Configuration UI must clarify "Athens time"**: Owner sets "Cancel up to 2 hours before appointment (Europe/Athens timezone)"; Greek UI: "Ακύρωση έως 2 ώρες πριν το ραντεβού (Ώρα Αθήνας)"
5. **Cutoff enforcement in cancelBookingWithRefund**: Add `cancellationCutoffHours: number | null` parameter (null = no cutoff); check **BEFORE** any ledger writes
6. **Add a Greek confirmation dialog** (as per milestone scope): Before cancel within cutoff window, prompt: "Σίγουρα; Χάνετε την κράτηση." Client explicitly confirms loss of credit

**Detection:**
- Enable logging with full context: `{ bookingDate, bookingTime, nowAthens, cutoffHours, decision, reason }`
- Set up alerts on DST transition dates (Oct 25, Mar 28) for cancellation-related errors
- Query: `SELECT COUNT(*) FROM bookings WHERE cancellation_forfeited = true AND calendar_date IN ('2026-10-25', '2027-03-28')` — unusual spike signals a bug

### Pitfall 4: Booking Mode Switch Orphaning Existing Bookings

**What goes wrong:**
1. Business runs in open_slots mode for 3 months; 200+ confirmed bookings exist
2. Owner decides to switch to fixed_sessions mode (configure recurring sessions via chat)
3. App updates `businesses.booking_mode = 'fixed_sessions'`
4. App creates session_catalog rows for next 12 weeks
5. A client tries to cancel an existing open_slots booking from 2 weeks ago
6. The booking row references `service_id` (in open_slots mode); new session-aware code expects `session_id`
7. `cancelBookingWithRefund` queries by service_id + time but doesn't find the session_id
8. Calendar sync, credit restore, and client notifications either orphan or break

**Alternative scenario:**
1. Business switches booking_mode from fixed_sessions → open_slots (reverting after trying sessions)
2. 50 pending session-based bookings exist
3. Bookings table has session_id FK that was populated
4. Open_slots mode assumes bookings link to services, not sessions
5. Queries fail: `SELECT * FROM bookings WHERE business_id = X AND session_id IS NOT NULL` — old bookings are visible to the wrong code path
6. Conflict detection (UNIQUE on (business_id, calendar_date, calendar_time)) doesn't account for session bookings; two clients accidentally book the same slot

**Why it happens:**
- The bookings table schema must support both modes simultaneously to avoid a complex migration
- Two queries interact with bookings: open_slots queries filter by `service_id + calendar_date + calendar_time`; fixed_sessions queries filter by `session_id`
- Without explicit mode scoping in queries, orphaned bookings become invisible or cause silent failures
- There's no "mode migration" step that re-indexes or validates existing bookings

**Consequences:**
- Silent failures on cancellation, leading to unreleased session credits
- Calendar deletion doesn't trigger (dangling Google Calendar events)
- Clients receive no confirmation of cancellation because the status update silently fails
- Owner's financial reports show phantom sessions or missing revenue

**Prevention:**
1. **Bookings table must have explicit session_id FK (nullable)**
2. **Mode switch must NOT update existing bookings** — leave them as-is: only update the mode flag, don't reindex
3. **Cancel/reschedule logic must detect mode + dispatch correctly**: detect `business.bookingMode`, log if cancelling old-mode booking in new-mode, proceed with same cancel logic
4. **Queries must respect mode without requiring schema migration**: listClientBookings filters on `session_id IS NOT NULL` (fixed_sessions) or `session_id IS NULL` (open_slots)
5. **Test mode switching explicitly**: Create bookings, switch modes, verify both old and new are queryable/cancellable, no booking lost or invisible

**Detection:**
- Query: `SELECT COUNT(*) FROM bookings WHERE business_id = X AND (session_id IS NULL AND booking_mode = 'fixed_sessions') OR (session_id IS NOT NULL AND booking_mode = 'open_slots')` — non-zero indicates orphaning
- Log every mode switch with before/after booking counts
- Alert if cancellation fails with "booking not found" but the booking exists in DB

### Pitfall 5: Telegram Rate Limit Cascade on Renewal Broadcast

**What goes wrong:**
Business has 150 active members nearing renewal. Owner decides to send a renewal nudge to all of them. App spawns a loop or parallel sends without throttling:

1. **Gemini rate limit**: 15 requests/minute on free tier. If each renewal message uses Gemini to render text, after 15 renewals, the 16th is queued/rejected
2. **Telegram rate limit**: Telegram's own per-chat limit (~30 messages/second per bot, with backoff). Rapid fire to 150 chats within 10 seconds triggers Telegram's anti-spam throttling
3. **Cascade effect**: If renewal broadcast is triggered by owner command (via AI tool), and each message is a Gemini call, tool execution phase stalls after 15 calls. Orchestrator retries, which re-fires broadcast, creating a cascading notification storm
4. **Client confusion**: Clients receive duplicate renewal reminders (1st attempt + retry)
5. **Database lock**: If broadcast loop writes to membership status table or calls membershipLedger inserts, `SELECT FOR UPDATE` lock on memberships can queue up transactions, causing bottleneck

**Why it happens:**
- Owner-triggered mass actions weren't anticipated in free-tier budget constraints
- Gemini free tier wasn't designed for batch operations
- The renewal feature says "broadcast" but doesn't specify throughput ceiling
- No rate limiter middleware exists (unlike sendTelegramMessage, which has retry logic but not backoff)
- Parallel `Promise.all()` is intuitive but violates Telegram's sequential delivery model

**Consequences:**
- Renewal notifications silently dropped (Telegram throttles after 30/sec)
- Clients miss renewal deadlines because some don't receive the message
- Owner sees partial success + unclear error states
- Service becomes unresponsive to other clients' booking requests (Telegram queue saturated)
- Database transaction queue builds up, cascading slowness

**Prevention:**
1. **Rate limit the broadcast loop**: Throttle to ~10 messages/second (well under Telegram's 30/sec)
2. **Don't use Gemini for renewal message rendering** — pre-compose templated messages server-side; only use Gemini for client-facing queries
3. **Broadcast runs in background job**, not critical path: Owner gets "Broadcast in progress" message, receives follow-up report when done
4. **Implement idempotency for broadcast**: Add `renewalBroadcast` table (business_id, broadcast_id, client_phone, status, sentAt, errorReason); resume from last successful index on retry
5. **Add per-user opt-out flag**: `clientBusinessRelationships.notifications_enabled`
6. **Monitor broadcast latency**: Log `{ broadcast_started, total_clients, send_time_ms, failures, success_rate }`; alert if success_rate < 95% or > 2h for < 150 clients

**Detection:**
- Query: `SELECT COUNT(*) FROM renewalBroadcast WHERE broadcast_id = X AND status = 'sent'` — compare against expected client count
- Log: Check for `error: 'rate_limit_exceeded'` from Telegram API responses
- Monitor: Track broadcast duration per-business; alert on > 30 min for < 150 clients

### Pitfall 6: Recurring Session Expansion Atomicity & Lazy-Loading Races

**What goes wrong:**
Owner creates recurring session: "Pilates Monday/Wednesday 18:00 for 12 weeks starting Oct 1". App attempts to:

**Eager Expansion (all at once):**
1. Generate 24 session_catalog rows (12 weeks × 2 days) inside single transaction
2. Insert all 24 atomically
3. If any INSERT fails (e.g., duplicate date), entire transaction rolls back; user sees vague error
4. Meanwhile, clients trying to book hit inconsistent state (partial expansion visible)

**Lazy Expansion (on-demand):**
1. Create `sessionRecurrence` row: (business_id, service_id, day_of_week, time, recurrence_end_date)
2. On each booking query, compute next N Mondays/Wednesdays; query `session_catalog LEFT JOIN sessionRecurrence`
3. Two concurrent booking requests both compute "next Monday" as available
4. Both try to INSERT session row for same date+time
5. First succeeds; second hits UNIQUE constraint

**Why it happens:**
- Eager expansion is atomic but creates large transactions (24+ rows) that can fail all-or-nothing
- Lazy expansion avoids large transactions but introduces time-of-check vs. time-of-use race
- Without application-level lock or DB-level `SELECT FOR UPDATE` on recurrence rule, both requests can race
- Booking UNIQUE constraint doesn't protect session_catalog from duplicate expansion

**Consequences:**
- Recurring sessions silently fail to expand; clients see "no availability"
- Duplicate session_catalog rows cause confusion
- Eager transactions block all other bookings (latency spike)
- Owner sees confusing error ("Transaction conflict")

**Prevention:**
1. **Use hybrid approach: eager expansion with idempotency keys**: Generate all rows, insert with `onConflictDoNothing()` on idempotency_key UNIQUE (pattern from Phase 7 `createMembership()`)
2. **Session catalog schema must have UNIQUE idempotency key**: both idempotency_key + unique_session_instance (business_id, session_date, session_time) constraints
3. **Bookings must block concurrent claims on same session**: Use `SELECT FOR UPDATE` on session rows (pattern from Phase 8 `getActiveMembershipForDeduction()`)
4. **For lazy expansion, use DB-level lock**: `sessionRecurrenceExpansionLock` table with row-level lock before expanding
5. **Test expansion race explicitly**: Create rule, spawn 3 concurrent booking requests, assert exactly 1 session + 1 booking succeeds

**Detection:**
- Query: `SELECT COUNT(*) FROM session_catalog WHERE business_id = X GROUP BY (session_date, session_time) HAVING COUNT(*) > 1` — non-empty indicates duplicates
- Query: `SELECT recurrence_rule_id, COUNT(*) FROM session_catalog WHERE recurrence_rule_id IS NOT NULL GROUP BY recurrence_rule_id` — compare COUNT against expected

## Moderate Pitfalls

### Pitfall 7: Incomplete Cancellation Cutoff Scope

Cutoff only enforced on client-initiated cancellations via `cancelAppointmentTool`, not on owner or system cancellations.

**Prevention**: Add `reason` parameter ('client_request' | 'owner_action' | 'system_expiry'); apply cutoff only to 'client_request'.

---

### Pitfall 8: Slotless Request Stale Data in Owner UI

Owner sees list of pending requests. One is from client whose membership expired 3 days ago. Owner taps "Approve"; app correctly rejects. But UI still shows "Pending." Owner confused.

**Prevention**: Filter requests before rendering: `WHERE status = 'pending' AND membership.expiresAt > now()`. Auto-refresh list every 60 seconds.

---

### Pitfall 9: Recurrence Deletion Cascading to Bookings

Owner deletes recurring pattern. App cascades delete to session_catalog, orphaning existing bookings.

**Prevention**: Use `ON DELETE SET NULL` or `ON DELETE RESTRICT` (not CASCADE). Require explicit cancellation of future sessions before deleting rule.

## Minor Pitfalls

### Pitfall 10: Session Capacity Counter Stale on Concurrent Cancels

Session capacity=1 (1 seat left). Two clients book concurrently → first succeeds, second fails. Both cancel simultaneously. Concurrent `UPDATE sessions SET capacity_remaining = capacity_remaining + 1` may leave counter at 1 instead of 2.

**Prevention**: Use `SELECT FOR UPDATE` pattern for capacity during cancellations as well as bookings.

---

### Pitfall 11: Renewal Broadcast Double-Sending on Network Retry

After 80/150 clients, network error occurs. Retry re-runs from 0. Clients 1-80 receive duplicates.

**Prevention**: Store broadcast state in DB. Resume from last successful index. Idempotency key per client.

## Integration Pitfalls

### Pitfall X1: Membership Expiry Sweep Racing with Approval

Expiry sweep marks membership inactive. Simultaneously, owner approves slotless request for same client. Approval reads membership, sees valid; sweep updates to inactive; approval inserts booking. Result: booking for expired membership.

**Prevention**: Both approval and sweep use `SELECT FOR UPDATE` on membership rows, serializing them.
